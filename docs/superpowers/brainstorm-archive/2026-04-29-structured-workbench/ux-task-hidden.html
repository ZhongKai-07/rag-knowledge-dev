"Task" 是后端 entity，不是 UI 概念

用户看到的是"知识问答 / 文档处理"两个能力区。点进去直接开干，不需要"创建任务"。

问题对照

❌ 错误：把 task 暴露给用户
Ops 工作台
➕ 开始新任务
步骤 1：点"开始新任务" →
步骤 2：选任务模板（COB / SOP / 筛查 / 提取） →
步骤 3：填任务名 →
步骤 4：终于能问问题了
用户体验问题：
· 多 3 步多余动作才能问一个问题
· "任务"是个技术词，用户不理解
· 比 ChatGPT 体验差太多
✅ 正确：task 隐藏在后端
Ops 工作台 | 💬 知识问答 | 📤 文档处理
💬 点"知识问答" → 直接看到聊天框，开问
步骤 1：点"知识问答" →
步骤 2：在聊天框里输入问题 ✅
后端发生的事：
· 用户提问的瞬间，自动创建一个 task instance
· 落 Ledger 时记录 task_type=knowledge_qa
· 用户左侧"历史"里看到"今天的对话"——这就是 task instance
· 用户从来不知道"task"这个词

用户语义 vs 系统术语 — 词汇映射表

用户看到的词 后端 entity 说明
💬 知识问答task_template = knowledge_qa能力区入口；点进去用知识问答 Agent
📤 文档处理task_template = doc_intelligence能力区入口；含筛查/提取两个子操作
📝 一段对话task_instance + conversation用户提问瞬间自动创建
🔍 一次筛查task_instance用户点"开始筛查"瞬间创建
📤 一次提取task_instance + extraction_job用户上传文档瞬间创建，可异步
"最近"列表task_instance 列表渲染就是用户的工作历史
🟢 COB / SOP chipkb_id 集合（task config）部门可访问的 KB 自动展开为 chip
核心原则：
  1. UI 上零"任务"二字——只有"知识问答 / 文档处理 / 最近 / 推送"
  2. 能力区即入口，点击直接进入对应交互（聊天 / 上传 / 选文档）
  3. task instance 自动创建——用户的第一个动作（提问 / 上传）触发，无感
  4. "最近"列表 = task instance 历史——用户能找回所有过去的工作
  5. COB vs SOP 在知识问答内做来源 chip——用户默认看到所有部门可访问 KB，可手动限定
对设计文档的影响：写 spec 时要严格区分"用户层 UI 文案"和"系统层 entity 命名"。前端组件命名也避开 Task* 前缀（用 ChatPanel / DocPanel / RecentList），后端再用 TaskInstance / TaskTemplate。
