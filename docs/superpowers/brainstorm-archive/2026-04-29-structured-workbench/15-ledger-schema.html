Ledger 双层 schema — 装下 5 类工作模式 + 不重构

五分法 / Self-improving / Insight / 评测自我迭代 — 全部依赖 Ledger 作为燃料。
Ledger schema 一旦定错,后面所有上层应用都要重构。这张图回答 — 一份 Ledger 表如何同时装下 5 类形态 + 留好未来扩展位。

① 为什么需要"双层"

常见错误做法:全部用 JSONB 一把梭。看起来灵活,实际上:
  • 查询不可索引 — Insight 想做"高频主题聚类",每次都全表扫
  • schema 漂移无约束 — 每个 Agent 自己决定 input/output 长什么样,半年后 5 类各有各的格式,跨任务聚合做不了
  • 评测 / 提议机制无法立足 — 信号筛选器要识别 bad case,需要结构化可查字段,不是 JSONB 大杂烩
正解 — 双层结构:
第一层:跨五类通用强类型列(所有 Agent 都填) — 索引可走、聚合可做、跨任务可比
第二层:每类独立 payload schema(JSONB + payload_type 强校验) — 类型特化分析走第二层

↓ 与 ↓ 共用一行 — 通过 payload_type 区分类型

第 2 层 · 分型 payload(每类独立 JSON schema)

写入时按 payload_type 强校验;每种 payload_type 对应 payload-schemas/{type}.json 资源文件
payload_type VARCHAR(32)knowledge_qa / doc_intel / cmd_exec / troubleshoot
payload JSONB类型 schema 由 payload_type 决定
关键约束:写入时必须注册过的 JSON schema 强校验。新增第 6 类时只需注册新 schema,旧表不动。

通用查询走第一层(不打开 JSONB);类型特化分析走第二层(每类各自字段已知)。两层各司其职 — 这就是"装下 5 类 + 不重构"的物理基础。

④ 三个非显然的设计点

① primary_*_text 是顶级列

Insight 的"高频主题聚类"和"bad case 排行"都需要直接读这两个字段;埋在 JSONB 里每个 Agent 自己定字段名 → 聚合时炸裂。这两列冗余存储但有索引 — 空间换 Insight 性能与一致性。

② sources_json 强制所有 Agent 填

这是 I4 可追溯不变量的物理落地。ArchUnit 规则:任何 Agent 写 Ledger 时 sources_json 必须非空(空数组也行,空数组表示"诚实地说没来源")。

③ user_feedback v1 列预留

试点 v1 不引入 thumbs UI(bad case 看 status=failed 就够),但列要先建 — v2 加 UI 时不补字段 / 不补 schema 演化。

⑤ 索引策略

-- 用户 own 路径:per-user task instance 历史 CREATE INDEX idx_ledger_user_created ON t_task_ledger (user_id, created_at DESC); -- Dept Insight:按部门 + 任务类型 + 时间聚合 CREATE INDEX idx_ledger_dept_template ON t_task_ledger (dept_id, task_template, created_at DESC); -- bad case 看板加速:partial index 仅索引非 success 行 CREATE INDEX idx_ledger_status_dept ON t_task_ledger (dept_id, status) WHERE status != 'success'; -- v2 thumbs 启用后查 feedback bad case CREATE INDEX idx_ledger_feedback ON t_task_ledger (user_feedback) WHERE user_feedback IS NOT NULL; -- 主题聚类:GIN 全文索引(v1 词袋,v2 升级向量) CREATE INDEX idx_ledger_intent_gin ON t_task_ledger USING GIN (to_tsvector('simple', primary_intent_text));

status != 'success' 的 partial index 是给 bad case 看板加速的 — v1 期间这是核心查询路径。
GIN 索引让"高频主题"的全文聚合走索引,不全表扫。

⑥ 双投影 — 同一份 Ledger 两个独立通道

技术投影(给 global_admin)

-- 系统健康看板:错误率 / 延迟 / 模型可用性 SELECT task_template, COUNT(*) AS total, SUM(CASE WHEN status='failed' THEN 1 END) AS failed, PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY duration_ms) AS p95, SUM(cost_tokens) AS tokens FROM t_task_ledger WHERE created_at > now()-'30 days'::interval GROUP BY task_template;

业务投影(给 dept_admin Insight)

-- 高频主题 top 5(v1 词袋粗聚类) SELECT primary_intent_text, COUNT(*) AS hits FROM t_task_ledger WHERE dept_id=:dept AND created_at > now()-'30 days'::interval GROUP BY topic_cluster(primary_intent_text) -- v2 升级向量聚类 ORDER BY hits DESC LIMIT 5; -- bad case 按失败模式聚合 SELECT task_template, payload->>'fail_subclass' AS subclass, COUNT(*) AS n FROM t_task_ledger WHERE dept_id=:dept AND status IN ('failed','partial') AND created_at > now()-'30 days'::interval GROUP BY task_template, subclass ORDER BY n DESC;

技术投影和业务投影读字段几乎不重叠 — 这就是 PA4(双级 admin 视图独立)的物理实现。一张表,两套 view,各看各的。

⑦ Ledger 在 v1 / v2 / v3 的演进

阶段Ledger 变更不变
v1 试点 新建 t_task_ledger 双层 schema · 4 种 payload schema 注册(2 种实际写入 + 2 种占位) —
v2 扩部门 启用 user_feedback 列(thumbs UI) · cmd_exec / troubleshoot 开始写入(对应 Agent 上线) · 加 dept_agg / global_agg view 表结构不变 · payload schema 不变 · 索引不变
v3 全员 主题聚类升级为向量(嵌入存储到 vector 列) · 加跨任务 / 跨文档 insight 视图 核心 schema 不变 · 历史数据兼容
v1 一次设计,v2/v3 不重构。每个阶段都是"加 view / 加索引 / 启用列",不是"改表"。
这是双层 schema 设计的根本价值 — 工程承载力。

⑨ 一句话总结

Ledger 双层 schema = "通用强类型列(走索引)+ 分型 payload(走 schema)"。
一张表装下五分法所有形态 · 通用查询走第一层 · 类型特化走第二层 · v1 全部 4 种 payload schema 注册到位 · v2/v3 接新 Agent 不改表。
这是 self-improving / Insight / 评测自我迭代三个上层应用能立足的工程基石。
