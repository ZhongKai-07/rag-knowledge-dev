两条业务闭环

用户体验闭环（个性化 + 主动触达）
① 用户问
小李：COB 问答 → "EU 区域的 MTA 默认值是多少？"
② Ledger 落账 + 隔夜抽取
Autodream 抽出：小李 = EU 业务 / 关注 MTA / 用 COB 多于 SOP
③ User Memory 形成
小李账号下：role=EU Ops / interests=[MTA, IA] / fav-tasks=[COB, Collateral 筛查]
④ 下次任务自动消费
小李第二天用 SOP 问答，Agent prompt 自动加："用户负责 EU 业务、近期关注 MTA"，相关 EU 章节优先
⑤ Pulse 推送
小李登录看到卡片：
🔔 "本周 EU 区域 MTA 政策更新（链接到新 chunk）"
🔔 "你昨天提取的协议 #1234 与历史 5 份 ISDA 对比异常"
🛡️ User Memory 用户可见 / 可编辑 / 可导出 / 可删除（合规）
部门运营闭环（insight → 反哺 KB）
① 全部门 Ledger 流
Ops 部门一周：3000 次 COB 问答 + 800 次 Collateral 筛查 + 50 份协议提取
② Autodream 聚类
发现：12 个用户问"EU MTA 阈值"——COB 手册没明确写 / 5 个用户问"协议提取后 X 字段为空"——schema 漏抽
③ 运营看板
🔥 本周热点 Top 10 / 🕳️ 覆盖盲区 / 📈 任务级 RAGAS 分布 / 🆘 反复问未解决 list
④ 反哺候选生成
LLM 自动生成：FAQ 候选条目 / COB 手册补丁草稿 / 协议提取 schema 优化建议
⑤ 人工审核 → 进 KB
Ops 主管在审核页一键通过 → 候选 chunk 标 source=llm_wiki 进 KB → 下一周用户问同样问题命中率 ↑
🛡️ 反哺必须经人审；llm_wiki 来源标记保留，可追溯可下线

跨任务、跨文档的"运营级 insight"（你提到的合同对比）

协议提取任务 → 部门运营 insight 的具体例子
原始事件流
协议提取任务运行 50 次
每次产出一份 Excel
Ledger 中存了 50 份 schema 化字段
Autodream 聚合
把 50 份提取结果
+ 历史合同 KB 中的 200 份
→ 全集字段对比分析
insight 卡片
"本月 50 份新签 ISDA 中：
· 3 份 MTA 高于 P95 历史值
· 5 份 IA 字段未填
· 2 份对手方角色异常"
关键点：这种"批量结果 → 跨文档 insight"是 PaaS 助手做不到的——它们各自孤岛。我们因为 Ledger 集中、schema 统一，聚合是免费的。

关键风险（与 roadmap §4 已有共识）

幻觉注入

  • LLM 生成 FAQ 候选 → 直接进 KB = 幻觉污染源
  • 缓解：人审强制闸门 + source=llm_wiki 标记 + 可追溯回原始 Ledger 行

成本放大

  • 每轮注 memory + 夜间批 + Pulse 生成 = LLM 调用数 ×N
  • 缓解：分级路由（抽取小模型，Pulse 大模型）+ 配额阈值 + kill switch

隐私 / PII

  • memory 自动捕获用户身份 / 项目 / 部门 → 泄露面扩
  • 缓解：可见 + 可编辑 + 可导出删除 + 敏感字段黑名单 + 跨 dept 不可见

质量漂移

  • 持续积累 ≠ 持续改进，可能反向腐化
  • 缓解：每个 Phase 上线前后跑 RAGAS 对比；任务级质量分进 OKR
