Self-improving 发动机 — 产品如何自己长大

"AI 原生工作台"区别于"AI 加壳的 SaaS"的核心,在于产品形态本身能进化。
机制 = 察觉(Ledger 信号) → 提议(规则引擎) → admin 一键决策 → 沉淀(产品形态进化)。

① 价值主张 — 业务永远只面对二元决策

核心承诺:业务用户(包括 dept admin)永远不需要思考 KB / Agent / Skill / Prompt / Schema 这些技术概念。

他们只需要在 inbox 看到:"系统建议你做 X,因为 Y" — 然后按一个按钮 Approve / Reject。
系统在背后自己沉淀部门记忆 / 提议新入口 / 清理死规则 / 派发 KB 补充工单。

 这是产品形态从"工具"到"操作中枢"的质变点 — 业务面对二元决策,产品在背后长大。

② 闭环流程

① 燃料
Ledger 落账(每次任务执行)
② 规则引擎
每天扫一次 · 5 类信号
③ 提议 inbox
证据 + 影响 + 30 天过期
④ admin 一键决策
Approve / Reject / 稍后
⑤ 沉淀生效
写 Memory · KB 工单 · 清理死条
↓ 下次任务执行受益于沉淀 → 数据回到 Ledger,新一轮闭环 ↓

核心是循环:每一个 admin 一键决策 = 产品真的进化一格。被拒绝的提议也有用 — 拒绝原因被记录,下次同类信号再生时附上"上次被拒的原因",系统也在学习什么时候该提议、什么时候不该。

③ 5 种提议形态 — 5 条产品进化路径

# 形态 信号源 沉淀到 v1 状态
F1 📚 沉淀部门记忆 ≥ 3 个用户 30 天内写过相似 User Memory t_dept_memory v1 做(规则)
F2 🆕 新 Task 卡片 ≥ 50 次 query 命中同 (KB 子集 + 主题词袋簇) 提议给开发(v1 仅展示) v1 雏形
F3 🔍 KB 补充候选 "未找到"模式 query 主题聚类 top N 派发 KB 补充工单(走人工 ingestion) v1 做(规则)
F4 ⚙ Agent prompt 调优 bad case 集中在某一类失败模式 改 Agent 内置 prompt(走开发 PR) v2(需 LLM 分析)
F5 🗑 部门记忆清理 某条 Dept Memory 60 天命中率 < 5% 软删除(30 天可恢复) v1 做(规则)
v1 试点期 = 3 种规则做 + 1 种雏形 + 1 种 v2 接。
不引入 LLM 抽提议(那是 v2 Autodream),但 5 种里有 3 种用规则就能跑起来,足够撑起试点期最小可用闭环。

④ 提议卡 — 永远三段式(建议 / 证据 / 影响)

📚 沉淀部门记忆 · 3 天前生成 30 天后过期
建议添加规则:"EUM = EU MTA"
▸ 证据:4 个用户在过去 30 天 User Memory 里写过相似术语
  • 用户 #3712 · "我说 EUM 都指 EU MTA" 2026-04-15
  • 用户 #2156 · "EUM = European MTA" 2026-04-18
  • 用户 #4901 · "EUM is EU MTA" 2026-04-22
  • 用户 #3712 · 工作台问答中使用过该简称 3 次
▸ 影响:批准后,所有部门成员的下一次问答 prompt 会自动注入此规则;用户问"EUM 是多少"时 Agent 自动转 query 为"EU MTA"。

三段固定:建议(说人话什么) · 证据(数据从哪来) · 影响(批了会怎样)。
让 admin 不是凭感觉拍板,是看数据决策;不是猜测后果,是知道点下去会发生什么。

⑤ 反学习机制 — 拒绝是有用的负反馈

📚 沉淀部门记忆 · 第 2 次提议 30 天后过期
⚠ 上次被拒原因:"这是某个用户的个人习惯,不是部门通用术语" · 本次新增 5 个用户支持,共 9 人使用
建议添加规则:"EUM = EU MTA"
▸ 证据(更新):9 个用户在过去 90 天使用该简称(原 4 人 + 新增 5 人) · 跨 3 个 sub-team
▸ 影响:同上次。批准后所有部门成员问答 prompt 注入此规则。
系统也在学:t_proposal_inbox.previous_rejection_id 引用上一次的拒绝。下次规则引擎扫到同主题信号时附上"上次被拒原因 + 本次新增证据"。

这条机制让"提议被拒"不是死路 — 它是一次校准。"被拒 → 新证据 → 再提议 → 可能批准"是 self-improving 在 admin 端最微妙的循环。

⑥ self-improving 在用户端的两个细微出口

出口 1 · 首屏"部门提醒"
─── 部门提醒 ────────────────
ⓘ 本部门最近有人在问"EU 区域 IA
  调整",要看吗?

过去 7 天部门有 ≥ 5 次同主题查询 + 当前用户没问过 → 显示。不打扰(无红点 / 仅 1 行)。让员工感到"AI 在背后看着部门趋势"。

出口 2 · "告诉 Lead 这块缺资料"
在 COB / SOP 里没找到。可能:
(1) 不在已上传文档
(2) 换个说法
(3) [告诉 Lead 这块缺资料]

点击 → 自动构造一条 KB 补充候选(F3)写入 t_proposal_inbox。业务的一次"找不到"自动变成 dept admin 的一条提议。

⑧ t_proposal_inbox schema

CREATE TABLE t_proposal_inbox (
  id                BIGSERIAL PRIMARY KEY,
  proposal_type     VARCHAR(32),    -- F1-F5
  target_dept_id    BIGINT,
  target_admin_role VARCHAR(16),    -- dept_admin / global_admin
  proposal_payload  JSONB,          -- 提议具体内容
  evidence_payload  JSONB,          -- 支持证据(涉及的 ledger 行/user 数/频次)
  status            VARCHAR(16),    -- pending / approved / rejected / expired
  reviewed_by       BIGINT NULL,
  review_note       TEXT NULL,
  previous_rejection_id BIGINT NULL,  -- 反学习引用
  expires_at        TIMESTAMP,      -- 30 天没处理自动 expired
  created_at        TIMESTAMP,
  reviewed_at       TIMESTAMP NULL
);

WHERE memory_type='dept'

⑨ 关键设计 — 不能做"全部批准"

不提供"全部批准"按钮。这不是性能问题 — 是产品价值。

批量一键违背 self-improving 的本质 — 每一条决策都是产品进化,需要 admin 看清证据/影响。
30 天没处理就过期(下次信号还在会再生),给 admin 喘息空间但保留压力。
批准后 30 天内可逆("已处理"屏的"恢复"按钮),避免 admin 拍错。
