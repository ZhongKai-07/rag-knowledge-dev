统一门户 · One Portal + 三态适配

前端越简单（One Portal），后端的意图路由（Intent Routing）和 RBAC 要求就越高。
统一入口不是"把系统缝合在一起"，而是"建立一个强大的智能调度前台"。

设计哲学

"AI 应当隐形于业务上下文之中。让应用找人，而不是人找应用。"

用户不需要认知"我用哪个 KB / 哪个 Agent / 哪个工作流"——业务意图触发后，系统自动选择合适的呈现形态。 这要求所有形态共享同一套领域驱动后台（DDD），仅在前端接不同的"入站适配器"（Inbound Adapter）。

P1 · 零技术词暴露 P2 · 第一动作即创建 P3 · 永远显示来源/置信度 P4 · 失败给具体下一步 P5 · Memory 透明不打扰 ★ 新增 P6 · 形态由意图决定，不由用户决定

② 同一前台 · 三种 AI 隐形形态

用户在主屏只看到一个搜索框。第一个动作（打字 / 拖文件 / 输入特定意图）后，意图路由决定走哪种形态—— 都是同一个用户视角的"我在工作"，没有"我在用 KB / 用 Agent / 用 Copilot"的认知切换。

③ 共用 DDD 后台 · 三种入站适配器

三种前端形态在后端共用同一套 Domain / Application Service / Agent / RBAC / Trace 链路； 差异只在 Inbound Adapter（如何接收用户输入、如何渲染响应）。

三种形态都自动落账，Memory 在 prompt 阶段注入
关键：IntentRouter 是新增层，不是改造现有 RAG。 现有 rewrite / retrieve / rerank 链路被包装进 KnowledgeQAAgent； IntentRouter 在更外层做"意图 → agent 组合 → 形态"决策。 这样三种形态可以独立演化，不互相耦合。

④ 与旧版（16-ux-home · 2 能力卡入口）对比

旧版：2 能力卡 + 入口决策

  • 用户进入 → 看到"💬 问知识 / 📤 处理文档"
  • 需要先决策"我要做哪种事"再进入
  • 入口数：2（+ 部门提醒 banner）
  • "协议筛查"硬塞到"📤 处理文档"内的二级菜单
  • 用户认知模型：能力 → 任务

新版：One Portal + 智能路由

  • 用户进入 → 看到一个搜索框
  • 动作即意图（打字 / 拖文件），形态自动确定
  • 入口数：1
  • "协议筛查"自然走 Mode B（Skill Card）路径
  • 用户认知模型：意图 → 系统选形态
↓ 决策点 -100% · ↑ IntentRouter 复杂度（受控）

⑤ 8 个关键设计决策

决策点决定理由
主入口 单一搜索框 + 拖拽 hint，不放能力卡 对齐 Glean / Notion 3.0 / M365 Copilot 趋势；让 AI 隐形于业务上下文
意图路由层 新增 IntentRouter 在 Application 层之上 把"形态选择"职责从用户挪到系统；与现有 RAG pipeline 解耦
形态 A · Chatbot 覆盖导航/咨询类意图（占预期流量 70%+） SOP/COB/政策类问题最自然的形态；现有 RAG 直接复用
形态 B · Skill Card 覆盖轻量审批/筛查，在 Chatbot 内动态吐卡 Collateral 准入需要"看 + 决策"，不能纯文本；卡片是结构化交付
形态 C · Doc Workbench 覆盖重度抽取/复核，独立路由 /docs/:taskId 对齐 iManage / ISDA Create 业内惯例；字段卡 + 原文高亮 + 置信度需要专属画布
形态切换可见性 形态 A→B 在同一对话流；A→C 跳路由但保留 task_id 上下文 A→B 是对话延续；A→C 是 modal 切换，URL 变更让用户感知"进入审核状态"
错路由兜底 低置信度 IntentRouter 输出 → 走默认 Chatbot + 顶部 chip 让用户切换 意图判错代价高（用户已说出问题但走错形态）；置信度阈值 + 用户可纠错
Cmd+K 命令面板 v1 不做（只在搜索框露 hint），v2 加 对齐 Notion 3.0；power user 的快捷通道，不占首屏视觉

⑥ 用户故事 — 张三的一天（三种形态自然切换）

9:15 · 形态 A 触发 张三打开工作台 → 主标题"今天，我能帮你做什么？"。 在搜索框输入"EU MTA 默认值是多少？" → IntentRouter 判定 = 咨询 → 进入 Mode A，给答案 + 来源卡。

11:30 · 形态 B 触发 收到一个新对手方咨询 → 直接在搜索框打"对手方 HSBC 能签 ISDA Master 吗？" → IntentRouter 判定 = 审批/筛查 → 进入 Mode A 但吐出 Skill Card（Mode B）， 展示对手方评级 + 限额 + 历史协议 → 张三点"同意准入"，记录入账，通知 Compliance。

14:00 · 形态 C 触发 收到 ISDA_HSBC.pdf 上传需求 → 张三直接拖文件到首屏 → IntentRouter 判定 = 文档处理 → 自动跳到 /docs/:taskId（Mode C）→ 左原文高亮、右字段抽取卡（带置信度）→ 张三复核 73% 的 MTA 字段后点"导出 Excel"。

17:30 · 都没有"创建任务" 张三全天没有点过"新建/上传/创建"任何东西， 没有思考过"我用哪个 KB / 哪个 agent"。他做的 3 件事都自然落账成 task instance，未来都可以"继续"。

这套 UX 让张三从来不需要思考：
❌ "我要进哪个页面？"（永远是首页搜索框）
❌ "这是个咨询问题还是审批问题？"（IntentRouter 判定）
❌ "我用哪个 agent？"（用户层根本不知道有 agent 概念）
❌ "怎么从问答切到文档处理？"（拖文件就行 / 系统自动跳路由）

⑦ 与现有 spec 的衔接

现有 spec 章节新增 / 修改
§3.1 三层 + 横切层 在 Application 层之上新增 IntentRouter 子层；不改现有三层职责
§3.2 Agent 接口 新增 ApprovalSkill（Mode B 用），契约保持 Agent<I, O>
§5.1 路由策略 主路由从 /tasks → /（One Portal）；新增 /docs/:taskId 给 Mode C
§5.2 工作台主界面 替换为本页 §1 的单输入框设计；旧 16-ux-home 归档为对比参考
§4 持续学习横切层 不动；三种形态都通过 TaskInstance 落账，Memory 注入点不变
§5.5 用户语义词汇映射 新增："形态 A/B/C" 是架构词，UI 从不出现；对用户只是"问问题 / 看建议 / 处理文档"
v1 落地优先级建议：
1. Mode A 全量上线（现有 RAG 直接包成 KnowledgeQAAgent + IntentRouter v0 = always-Chat）
2. Mode C 试点 ISDA 提取（独立路由 + DocIntelligenceAgent，不依赖 IntentRouter 完美准确度）
3. Mode B 后置（依赖 ApprovalSkill 框架 + 业务方对"准入卡"字段达成共识）
4. IntentRouter 智能化（v1 用规则 + 文件嗅探，v2 上 LLM 分类，v3 加用户偏好学习）
