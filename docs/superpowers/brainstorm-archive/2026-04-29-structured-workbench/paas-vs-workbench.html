不是"我们做得更好的助手平台"，是不在同一层。复用单位错了，再好的拖拽工具也救不回来。

作坊问题的根因（不是工具不好）

关键洞察：一个助手 ≈ 80% 配置（prompt / 语料 / 工具列表 / UI 形态）+ 20% 代码（编排逻辑 / 引用规则 / eval / 防护栏）。

PaaS 把这两部分捆在"助手"一个东西里，鼓励"每个助手 100% 自己造"——结果配置和代码都被复制。

我们的做法 = 切开：
  • 20% 代码 → 共享 Agent 库（少数、精炼、有 owner、统一升级）
  • 80% 配置 → 任务定义（多、轻、未来甚至可由 LLM 辅助生成）
这一刀下去，"低水平重复"才被真正切掉。

类比

PaaS 助手平台 ≈ Wordpress 主题市场

  • 每个网站 fork 一份完整模板
  • 主题改自己的，主题间无机制共享
  • 插件市场看似丰富，实际良莠不齐、维护各异

结构化工作台 ≈ 操作系统 + 应用

  • OS 提供能力（文件 / 网络 / UI 框架）
  • 应用消费 OS 能力，不重写
  • OS 升级 → 所有应用自动受益
  • 新应用门槛低（写配置），新能力门槛高（要进 OS 评审）
我们的"反作坊"承诺（产品价值主张）：
  • 📦 语料零重复——所有任务共享底座 KB，永不复制上传
  • 🛡️ 权限零旁路——任务自动继承底座 RBAC + security_level
  • 📊 质量可对比——RAGAS 评测在所有任务统一接
  • 🔄 升级零迁移——Agent 升级，所有依赖任务自动受益
  • 🚫 无幽灵助手——任务有 owner、有评审、有下线机制
诚实的反向风险：这套要成立，必须有产品/架构纪律——一个负责人能在"再加一个 Agent" vs "复用现有 Agent + 加 skill"之间 say no。否则 Agent 数量也会爆炸，退化回 PaaS。这是治理问题，不是技术问题。
