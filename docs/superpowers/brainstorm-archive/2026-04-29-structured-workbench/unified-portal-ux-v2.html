② v1 vs v2 关键变化

维度 v1（极简版） v2（修正版） 理由
主入口 仅 1 个搜索框 搜索框 + 上方 7 个 Skill chips chips 是 affordance 不是决策点；不点也能用
任务列表 底部 inline "最近"（3 条） 左侧栏 first-class · 项目分组 对齐 Code-Buddy/悟空；项目分组让多周期任务可管理
新建动作 隐式（敲字即创建） 左上"＋ 新对话"显式按钮 + 仍支持隐式 显式按钮让"清空当前 / 开新会话"更直觉
Mode/Project 无 输入框下"智能"+"EU Ops"两个 pill 对齐 Code-Buddy 的 Agent+Model；用户可显式锁定项目上下文
Skill 发现 仅 IntentRouter 后端隐式 chips 显式陈列 + IntentRouter 仍兜底 新用户不点 chip 永远不知道能做什么；这是 onboarding 致命问题
项目分组 无 侧栏按项目折叠（EU Ops / APAC Ops / 合规专项） 对齐悟空"项目"概念；Ops 用户跨多对手方/区域工作时必需
二级功能区 无（仅头像下拉藏 Memory） 侧栏"技能中心 / 定时任务 / 收藏" 给 v2 留口子（定时任务 = Autodream 出口）；v1 不必上线但 IA 留位
主屏决策点 1 个（输入框） 仍是 1 个（chips/sidebar 都是 affordance/历史） One Portal 哲学守住
关键判断：v2 看起来"更密"但决策点没增加。
用户主流程仍然是"打字 / 拖文件" → Mode A/B/C 由 IntentRouter 决定。 Skill chips 是横向 affordance（让用户知道系统会做什么），点 chip 等价于在搜索框输入对应 prompt。 Sidebar 是历史和项目结构，不是入口（用户不用先去那里"创建"）。

③ "三件套"对应到我们的 DDD 架构

前端三件套 UI 表现 后端 entity v1 落地
Chatbot 入口 主屏单一搜索框 + 拖拽 hint IntentRouter · 路由到 Agent 组合 规则版：文本→KnowledgeQAAgent / 文件→DocIntelligenceAgent
任务为单元 左侧栏任务树 + 项目分组 + 最近列表 TaskInstance · 自动创建 / 项目 tag / 用户可重命名 沿用 spec §3.2 设计，新增 project_tag 字段（可空）
Skill 库 chips 横向 + 技能中心二级 TaskTemplate · 每个 chip = 一个内置 template v1 = 7 个内置 chip；点击 = 把 template 的 prompt prefill 到输入框
这是同一套 DDD 后台的三种前端入口：
  • 用户从 chip 点进 = TaskTemplate prefill → IntentRouter 直接命中 → 进对应 Mode
  • 用户从输入框打字 = IntentRouter 自由路由 → 命中或兜底 Chatbot
  • 用户从侧栏点旧任务 = 加载 TaskInstance 上下文 → 继续
三条路径共用同一套 Agent / RBAC / Trace / Memory 链路。

④ 设计原则更新（P6 + P7 新增）

P1 零"任务 / 配置 / Agent / Skill"字眼 — UI 对用户用"对话 / 技能 / 项目"等业务词
P2 第一动作即创建，空状态不入库
P3 永远显示来源 / 引用 / 置信度
P4 诚实失败用主动语态，给具体下一步
P5 个人 / 部门 memory 透明但不打扰
P6 形态由意图决定，不由用户决定（IntentRouter 做决策）
★ P7 显式 affordance ≠ 决策点。Skill chips 让用户"知道能做什么"，但不强迫先决策；点 chip = prefill 到输入框，仍走单一路由

⑤ 待你定的几个细节

问题选项
Skill chips 数量 v1 给 5 个还是 7 个？给少了 affordance 弱，给多了视觉密；建议起步 5（COB 咨询 / SOP 流程 / 协议筛查 / ISDA 提取 / 准入审批）
项目分组来源 (a) 用户手动创建项目；(b) 系统按 KB 自动分；(c) 按对手方/区域自动 tag。建议 (a) 起步，v2 加 (c) 自动建议
"技能中心"页面 v1 是否上？建议 v1 仅在侧栏留入口但点击只显示 toast"v1.5 上线"，避免页面空跑
"定时任务" v1 完全不做，但侧栏留位置（呼应 Autodream 路线）
左侧栏可折叠？ 建议默认展开，提供顶部 ⊢ 折叠按钮（对齐 Code-Buddy / 悟空都有）
"新对话" vs "新任务" UI 用"新对话"（用户语义），后端 entity 仍是 TaskInstance（架构语义）— 符合 P1

⑥ 与现有 spec 衔接

本页相对 v1（unified-portal-ux.html）和 spec §5.2 的主要修改点：
  • 新增左侧栏（任务树 + 项目分组 + 二级功能区）→ spec §6.1 前端组件需新增 SidebarPanel
  • 新增 Skill chips 行 → spec §6.1 新增 SkillChipsBar + 后端 TaskTemplate 暴露 chip 元数据接口
  • 输入框下 Mode/Project 切换器 → spec §3.2 IntentRouter 接受 user override（强制走某 Mode）
  • 原则 P7 新增 → spec §5 用户语义词汇映射表加入"affordance vs 决策点"约束
  • 路由仍是 /(One Portal) + /docs/:taskId(Mode C)，不变
