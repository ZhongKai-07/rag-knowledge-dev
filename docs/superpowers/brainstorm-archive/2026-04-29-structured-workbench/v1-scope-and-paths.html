v1 范围锁定 + 3 种实施路径

范围已对齐。剩下一个关键决策：怎么落地到当前代码库，决定工作量、风险、迁移代价。

3 种 v1 实施路径（关键决策 ⚠️）

A · 渐进改造（保 KB 入口）
/spaces 现状不变（KB 列表）
⊕ /tasks 新入口（4 个任务卡片）
/chat 共用
后端：抽 task / agent 抽象，旧 RAG controller 接到新 Agent

A · 渐进改造

新旧入口并存。Ops 任务走 /tasks，老用户继续 /spaces。低风险但概念混乱。

B · 新工作台并存 ⭐
/workbench 新主入口（4 任务卡）
/spaces 移到管理后台（KB CRUD）
/chat 退役 → 任务内嵌入
后端：新 task / agent 模块；旧 controller 标 deprecated 不删

B · 新工作台并存 ⭐

新前端入口 = 工作台。KB 概念回退到管理员视角。Ops 用户只看任务，不再"先选 KB 再聊天"。中等工作量、概念清晰。

C · 一次性切换
/workbench 替代所有现有路由
/spaces /chat 全部下线
所有交互都走任务模板
后端：旧 controller 同步删除，无并存期

C · 一次性切换

干净彻底，没有遗留概念。但前端工作量大、回归风险高、灰度难。

3 种路径对照

维度 A 渐进改造 B 新工作台并存 ⭐ C 一次性切换
前端工作量小（加 1 入口）中（新主入口 + 旧路由保留）大（重写所有交互）
后端工作量中中（旧 controller 不删）大（要同步迁移所有调用方）
概念清晰度差（KB+任务双概念并存）好（KB→管理；任务→用户）最好
回归风险低低-中高
灰度能力好（旧入口兜底）好（feature flag 切主入口）差（无回滚路径）
对持续学习层两套埋点（旧 + 新）新路径埋全单套埋点
v2 演进难度难（旧入口要持续兼容）易（旧入口逐步下线）最易
我推荐 B（新工作台并存），理由：
  • 概念清晰：Ops 用户进 /workbench 只看到任务，不需要"先选 KB"。KB 退回管理员视角，回归"语料管理"的本质身份。
  • 风险可控：旧 /spaces/chat 路由保留作为兜底，feature flag 控制 /workbench 主入口，可灰度。
  • 持续学习层埋点干净：所有 v1 任务都走新 task / agent 链路，Ledger 记录统一，不会有"旧路径漏埋点"的脏数据。
  • v2 演进顺：当 4 个任务都跑稳了，旧路由 deprecate 一个版本就能下线，不用反复迁移。

反对意见：B 比 C 多一份"过渡期"代码债。但相比 C 的回归风险，这个债是可控的。
