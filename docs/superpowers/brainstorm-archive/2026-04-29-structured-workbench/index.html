结构化工作台 v1 设计 — Brainstorm 过程归档

以下 6 个 mockup 是设计文档（spec）落地前的探索过程，按对话推进顺序排列。Spec 是结论，这里是结论的来路。

背景：2026-04-29 与产品方对齐 Ops 部门的 4 个 AI 应用场景（COB / SOP / Collateral 筛查 / 协议提取）的产品形态。最终决策落到 docs/superpowers/specs/2026-04-29-structured-workbench-design.md。

探索顺序

1️⃣
产品形态选择

KB 中心化 vs Chatbot 集合 vs 结构化工作台

三种产品形态对比，最终选 C：结构化工作台。
2️⃣
抽象层级

Task vs Agent+Skill 不是二选一

用户层是任务，系统层是 Agent+Skill，两层都要。
3️⃣
差异化 vs PaaS 作坊

为什么不是 agent 拖拽平台

PaaS 助手平台问题在复用单位错了，不是工具不好。
4️⃣
持续学习横切层

Self-Improving + 知识运营

Ledger 为底盘的 self-improving + 部门 insight 闭环。
5️⃣
v1 范围 + 实施路径

4 任务 + 2 Agent + 最小闭环

3 种实施路径对比，最终选 A：渐进改造（试点期保留新旧并存）。
6️⃣
UX 术语锁定

Task 是后端 entity 不是 UI 概念

用户看到的是"知识问答 / 文档处理 / 最近"，不是"任务"。
关键决策落点（在 spec 里继续展开）：
  • 三层架构：用户层（Task）+ Agent 层 + 底座，加横切持续学习层
  • v1 范围：4 任务 / 2 Agent / Ledger 多任务 schema / User Memory v1（用户主动声明）
  • 实施路径 A：新增 /tasks 入口，老 /spaces 兜底，试点期可比对
  • UX 术语：UI 上零"任务"二字；TaskInstance / TaskTemplate 仅存活在后端代码
