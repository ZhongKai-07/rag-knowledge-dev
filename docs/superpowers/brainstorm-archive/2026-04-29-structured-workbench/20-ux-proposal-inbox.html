屏 6 · 系统提议 — self-improving 在 admin 端的物质化

业务管家永远只面对二元决策(Approve / Reject) — 产品在背后自己沉淀。

② 三段式信息结构(每张提议卡通用)

段内容价值
建议 说人话的 Action(添加规则 X / 补充 Y 资料 / 下线 Z) 让 admin 立刻知道"该做什么"
证据 数据从哪来 — 涉及的 user 数 / query 数 / 命中率 + 可下钻看具体 Ledger 行 让 admin 不是凭感觉拍板,是看数据决策
影响 批准 / 拒绝后会发生什么 — 用"下次问答"现实语义,不用"prompt 注入"技术词 让 admin 知道点下去后果是什么
关键 UX 设计:不是"AI 给你列了个建议清单",而是"系统观察到现象 X(证据),建议 Y(行动),会带来 Z(影响)"。三段缺一不可,缺一段就退化成"普通管理后台",失去 self-improving 的差异化体感。

③ 反学习机制 — 拒绝原因驱动的二次提议

系统也在学:t_proposal_inbox.previous_rejection_id 引用上一次拒绝。下次扫到同主题信号时附"上次被拒原因 + 本次新增证据"。

例子(屏 6 第一张提议):
第 1 次:4 个用户用 EUM 简称 → 系统提议沉淀部门 → admin 拒绝"个人习惯不是部门通用"
第 2 次(60 天后):新增 5 人,共 9 人,跨 3 个 sub-team → 重新提议,显式提示"上次拒绝原因"+ 本次更强证据 → admin 此时大概率批准

 "被拒 → 新证据 → 再提议 → 可能批准"是 self-improving 在 admin 端最微妙的循环 — 拒绝不是死路,是校准。

⑤ 关键设计 — 不能"全部批准"

不提供"全部批准"按钮。这不是性能问题 — 是产品价值。

批量一键违背 self-improving 的本质 — 每一条决策都是产品进化,需要 admin 看清证据/影响。

设计配套:
• 30 天没处理就过期(下次信号还在会再生),给 admin 喘息空间但保留压力
• 顶部 tab 按类型切,admin 可以按精力批处理(集中处理"部门记忆"或"清理")
• "稍后"按钮把当前条目推到下次进入时再看
• 批准后 30 天内可逆(已处理屏的"恢复"按钮)— 避免 admin 拍错

⑥ 横切设计点

点设计
数据刷新每天凌晨 3 点跑提议规则引擎;当天新增提议在屏 6 顶部新条目

移动端v1 不做
提议数量上限同类型同主题只生成 1 条 pending 提议;30 天过期清理
误操作批准即时生效,但通过"已处理"屏的"恢复"按钮 30 天内可逆

⑦ UX ↔ schema

UI 元素schema 字段
提议卡(三段式)proposal_payload(JSONB) + evidence_payload(JSONB)
"30 天后过期"expires_at
"第 2 次提议"标签previous_rejection_id 非空
"上次被拒原因"bannerJOIN previous_rejection 取 review_note
批准UPDATE status='approved' + 触发对应沉淀动作(写 t_dept_memory / 派发 KB 工单 / 软删 memory)
拒绝UPDATE status='rejected' + review_note 必填
30 天可逆已处理项 status = 'approved' AND reviewed_at > now()-30d → 可恢复

⑨ 用户故事 — 一次提议被拒后的反学习

第 1 次(60 天前):系统检测到 4 个用户在 User Memory 写过 "EUM = EU MTA" 类似事实 → 生成提议 → 李四看到后觉得"4 个用户里有 2 个是同一个 sub-team,可能是局部习惯" → 拒绝,写原因"个人习惯,不是部门通用"

第 2 次(今天):60 天过去,系统再次扫到 — 这次 9 个用户,跨 3 个 sub-team 都在用 → 生成第 2 次提议 → UI 显著标注"第 2 次提议" + 上次被拒原因 + 本次新增证据

李四:"上次我担心是局部习惯,但现在跨 3 个 team 都在用,确实该入部门规则" → 一键批准

沉淀:t_dept_memory 多一条 "EUM = EU MTA" → 部门所有人下次问"EUM 是多少"时 Agent 自动识别为"EU MTA" → 不再有人提"EUM 不被识别"的痛点

这就是"产品在背后自己长大"的具体形态 — 李四只做了一次"批准",系统自己完成了证据收集 / 反学习 / 沉淀 / 注入。
Reject 不是死路,是给系统的负反馈让它"再观察 60 天" — 这种产品形态在传统 SaaS 里是不存在的。
