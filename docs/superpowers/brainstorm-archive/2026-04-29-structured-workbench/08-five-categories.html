从"AI 作坊"到"智能工厂"——场景的五分法

PaaS 助手市场看似遍地开花，但绝大部分助手是"高级定制款"。每个场景从需求对接到 Prompt 调优、工具集成、效果验证都是信息孤岛。
要走出这个困境，需要把"按场景切片"换成"按工作模式切片"——这就是五分法的由来。

① 困境：AI 作坊的根因不是工具不好,而是切片角度错了

PaaS 助手作坊
按场景切

每个业务场景独立造一个助手 — MAC 核对、单测、排障、催收……

  • 16 个助手 = 16 份独立 prompt + 语料 + 工具 + eval
  • 同一份制度文档被 5 个助手各自上传 5 份
  • 没人能回答"底层有几种本质能力"
  • 升级模型 = 16 个助手挨个改
  • 用户找不到入口("哪个助手能做我这事?")
→
智能工厂
按工作模式切

先观察日常工作的共性,沉淀 5 类底层工作模式,再用少数几个 Agent 覆盖 80% 任务。

  • 新场景 = 写一份配置(不写代码)
  • 语料统一在底座,零重复
  • Agent 升级 = 所有依赖任务自动获益
  • RAGAS / 引用约束等"非作坊"基建天然共享
  • 新增任务的边际成本递减
第一性原理:工作内容五花八门,但近 80% 的任务可以被归为四大类工作模式 + 一类碎片化兜底。
— 不是凭空抽象,是对日常工作"贴身观察"和行为分析后的归纳。

② 五分法 — 4 类主干 + 1 类兜底

类别 本质特征(工作流骨架) 典型场景 v1 状态

③ 五分法成立的四条硬条件

一套场景分类法要能撑起"智能工厂",必须同时满足下面四条;缺一条就退化成"事后归纳的标签"。

#条件含义不满足时的症状
C1 正交性 一个场景只能落进一类,落不进两类 出现"既是 A 又是 B"的灰色场景 → 分类粒度错
C2 完备性 80% 任务能落进 4 类(碎片化兜底剩余 20%) 高频场景被迫塞进"碎片化" → 分类太窄
C3 同类同构 同类内不同场景的"工作流骨架"几乎一样 同类两个场景骨架差很多 → 分类太宽
C4 跨类可组合 跨类任务能用"Task 编排层"组合解决,不需要造第 6 类 总有"四不像"任务 → 分类的维度选错了

④ Ops 4 场景压测结果

Ops 场景落在哪类C1 正交C3 同构备注
COB 问答 知识问答 ✅ 干净 ✅ 与 SOP 同构 第 1 类的明确锚点
SOP 问答 知识问答 ✅ 干净 ✅ 同上 同上
协议提取 文档智能 (extract_full) ✅ 干净 ✅ ISDA/GMRA 同构 第 2 类的明确锚点
Collateral 智能筛查 文档智能 (targeted_screen) ✅ 同 Agent 内 skill 切分 — 不是跨 Agent;是 DocIntelligenceAgent 的另一个 skill

Ops 4 场景验证了第 1 + 2 类。第 3 + 4 类 Ops 当前没有真实场景(复杂指令需业务系统 API 开放;问题排查更多是 IT 场景),但骨架清晰,值得在 v1 预留接口。

⑤ v1 试点工程范围 = 2 实现 + 3 预留 + 1 兜底

v1 实际实现的 Agent
💬 KnowledgeQAAgent
包装现有 RAG pipeline。服务 COB / SOP 问答。
📤 DocIntelligenceAgent
两个 skill:extract_full(协议提取) + targeted_screen(协议查询)
v1 预留接口 / v2-v3 实现
⚙ ToolOrchestrationAgent
等业务系统 API / MCP 开放后实现。
🔍 TroubleshootAgent
v2 候选;承接 IT 自助答疑场景。
· 碎片化兜底
不专门建 Agent,通用聊天承接长尾。
"预留"的具体含义:不是写一份 stub class — 是 v1 的 Agent 接口、Task 编排层、底座横切能力(工具治理/策略配置/评测闭环)必须能在 v2 接两类新 Agent 时不需要重构。这是分类法的真正价值 — 给未来留好接口位,避免每次新场景都"急造"。

⑥ 治理硬约束 — Agent 数量上界 = 5

红线:Agent 数量 > 5 = 切片角度又错了,需要重新归纳分类法,而不是再加一个 Agent。

新增 Agent 必须走架构 review;ArchUnit 规则限制新 agent 包必须实现 Agent 接口;任意 PR 增加超过 1 个 Agent 必须 spec。这是治理问题,不是技术问题。
