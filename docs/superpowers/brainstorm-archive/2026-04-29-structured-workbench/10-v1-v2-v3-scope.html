v1 / v2 / v3 切分 — 试点先行

五分法 + 三件事 + 双级 admin + 持续学习 — 看上去野心大,但 v1 工程范围严格收敛。
试点先在 Ops 单部门跑通最小闭环,再扩部门、扩能力。每一阶段都接已留好的接口,不重构。

① 三阶段定位

阶段

v1 · Ops 单部门试点

目标:跑通最小可用闭环。验证产品形态、收集数据、验证 self-improving 闭环。

  • 2 个 Agent 实现 + 3 个预留接口 + 1 个兜底
  • 双级 admin 雏形(global + dept)
  • Memory + Insight + 3 类规则提议
  • 新旧入口并存(/workbench 主推 + /spaces 兜底)
下一阶段

v2 · 扩部门 + 工具/排查

目标:扩到 2-5 个部门。加复杂指令 / 问题排查 Agent。

  • 多级 admin 完整(KB / Agent admin)
  • 工具治理流水线 + 工具风险分级
  • Autodream 自动抽 Memory + 集体记忆
  • thumbs-up/down + 字段纠正信号
  • /spaces 下线
远期

v3 · 全员 / 跨域

目标:全公司部门级铺开。AI 反哺人类知识库。

  • 跨部门授权 / OPA / OpenFGA
  • 反哺 KB(LLM-wiki source)
  • Pulse 推送(站内卡片/邮件/IM)
  • 跨任务 / 跨文档 insight
  • 多端(移动 / IM 集成)
关键判断:v2 / v3 的所有增量 = "接已留好的接口",不是重构。
ArchUnit 规则 + Agent 接口 + Ledger schema(双层) + 提议引擎 + Memory 注入钩子 — 全部在 v1 落地时就预留好了 v2/v3 的扩展位。每一阶段的工程改动是"加,不是改"。

② v1 工程范围 = 2 实现 + 3 预留 + 1 兜底

类别 v1 状态 具体落地内容 背后真实场景驱动
💬 知识问答 v1 实现 KnowledgeQAAgent · 包装现有 RAG pipeline · 服务 COB / SOP 问答 Ops 部门已有制度文档咨询需求
📤 文档智能 v1 实现 DocIntelligenceAgent · 两个 skill:extract_full(协议提取) + targeted_screen(协议查询) ISDA / CSA / GMRA 协议入档 + cloudmarging 录入
⚙ 复杂指令 接口预留 Agent 接口可注册一个未来的 ToolOrchestrationAgent · ArchUnit 规则就位 等业务系统 API / MCP 开放后实现
🔍 问题排查 接口预留 Agent 接口可注册 TroubleshootAgent · ReAct 推理框架预研 v2 候选(IT 自助答疑场景)
· 碎片化兜底 隐式存在 不专门建 Agent · 通用聊天承接长尾问题 —

③ 横切底座的四块 — v1 必做哪些

横切底座 v1 范围 v1 具体落地
1. KB · RBAC · Trace · RAGAS 现状沿用 OpenSearch 主路径 / KbAccessCalculator / RagTraceContext / 现有 t_eval_* 表
2. 工具治理 / 入驻链 v2 实现 v1 不实现;复杂指令 v1 不实现,工具治理可推
3. 管理员策略层 部分 v1 + v2 完整 v1: 双级 admin 雏形(global + dept) + 简单字段集维护;v2: 多级 admin / 推理策略 / 话术映射
4. 持续学习层 v1 必做 Ledger 双层 schema + User Memory + Dept Memory + Insight 看板 + 3 类规则提议 + promote 流程
试点期权重:横切底座的"持续学习层"是 v1 重头戏,占工程量 ~40%。"工具治理"和"管理员策略层"绝大部分推到 v2 — 试点期 Ops 单部门内部默认互信,先验证产品形态,不深耕治理。

⑤ v1 工程改动量盘点

分类数量明细
新增数据库表 ~7 张 t_task_ledger(可重建) · t_user_memory · t_dept_memory · t_memory_promote_request · t_memory_injection_log · t_insight_audit_log · t_proposal_inbox
新增 service ~15 个 TaskTemplateRegistry / TaskInstanceService / AgentRegistry / LedgerWriter / SignalFilter / ProposalEngine / UserMemoryService / DeptMemoryService / MemoryInjector / InsightAggregator / PromoteService / AuditLogger / etc.
新增前端页面 ~9 屏 /workbench(屏 1) · 知识问答(屏 2a) · 文档处理(屏 2b 含两子屏) · /profile/memory · /admin/insight(屏 3) · /admin/dept-memory(屏 4) · /admin/memory-inbox(屏 5) · /admin/proposal-inbox(屏 6) · /admin/dept-fields(屏 7)
新增 ArchUnit 规则 ~5 条 workbench 不得 import 其他域 mapper / agent 不得 import knowledge.dao 或 rag.dao / 新 agent 必须实现 Agent 接口 / Ledger 写入 sources_json 必须非空 / 异步路径不读 UserContext
YAML 资源文件 4 份 schema isda.yaml / csa.yaml / gmra.yaml / isda+csa.yaml(每份含字段定义 + aliases + examples)

改动面看似多,但每一项都对应明确的产品价值点。没有"为重构而重构"的改动;现有 RAG pipeline / KB CRUD / RBAC 链路完全沿用,不动。

⑥ 渐进改造的双轨期

v1 期间 /spaces 与 /workbench 并存:前者是老体验、后者是新体验。

可通过看板对比两套入口的活跃度 / 任务完成率 / RAGAS 分数,作为 v2 决定旧入口下线时机的依据。
顶部导航栏加"Ops 工作台"链接到 /workbench,并在 /spaces 页面顶部加一条引流条("试试新版工作台 →"),不强制跳转。

⑦ v1 整体验收标准

#验收项
1/workbench 主入口可访问,不需要"创建任务"动作即可开始问答 / 上传
2COB 问答结果与现有 /chat 在同一 query 下答案一致性 ≥ 95%(RAGAS 对照)
3协议提取(ISDA)字段准确率 baseline ≥ 80%
4协议查询(targeted_screen)实体识别准确率 ≥ 85%
5任意一次任务执行后 t_task_ledger 有对应行 + trace_id 串通
6所有 Agent 产出 sources_json 必填非空(I4 可追溯不变量)
7/profile/memory 用户可手填 / 编辑 / 删除 memory,下次提问 prompt 含 memory 段
8所有任务执行受现有 RBAC + security_level 拦截,零旁路
9ArchUnit 新规则全部 green
10UI 文案 review 通过:零"任务/任务模板/任务实例/Agent/Skill"等技术词
11/spaces + /chat 老路径在 v1 内仍可用(兜底)
12提议引擎跑通:30 天内至少生成过 3 类提议各 1 条;反学习字段 previous_rejection_id 串通
13双投影:dept admin 看 Insight / global admin 看系统健康 — 两者数据视图不混合

⑧ 关键风险与缓解

风险严重度缓解
架构纪律失守 → Agent 数量爆炸退化为 PaaS 高 新增 Agent 必须走架构 review;ArchUnit 规则限制;PR 增加 > 1 个 Agent 必须 spec
协议提取/查询准确率不达标 中 v1 限定 4 种 schema · 字段级评测 baseline ≥ 80%
试点扩部门时数据污染 中 Ledger visibility_scope 字段已建好;v1 默认 DEPT;v2 加 view 即可分层
Memory 注入 prompt 失控(爆 token) 低 MemoryInjector 限定 ≤ 500 token;超过截断
PII 隐私(用户主动声明形态较安全) 低 v1 全部 user-visible / user-editable / user-deletable
