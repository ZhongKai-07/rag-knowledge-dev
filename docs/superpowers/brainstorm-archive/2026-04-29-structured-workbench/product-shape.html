产品形态：用户进入系统第一眼看到什么？

三种形态对后端 pipeline、任务抽象、UI 都有根本不同。先选大方向，再聊实现。

A. KB 中心化（≈ 当前）
HT KnowledgeBase | 我的KB | 上传
📚 COB 制度库
📚 SOP 操作手册
📚 Collateral 协议库
📚 + 新建 KB
点 KB → 进入聊天
💬 Q&A 聊天框（所有 KB 都长这样）

A · KB 中心化

不同语料 = 不同 KB。所有交互形态 = 同一个 RAG 聊天框。协议提取无家可归。

B. 任务模板化 Chatbot 集合
Ops 助手 | 选择任务
🤖 COB 咨询
🤖 SOP 助手
🤖 Collateral 筛查
🤖 协议提取（？）
每个 chatbot = 聊天形态 + 绑定语料/prompt/工具
💬 还是聊天框，但带"任务 hint"

B · 任务模板化 Chatbot

每个任务 = 一个绑定语料/prompt/工具的 chatbot。统一是聊天形态。协议提取硬塞会很别扭。

C. 结构化工作台（任务驱动）
Ops 工作台 | 任务中心 | 任务历史
💬 COB 问答
→ 聊天 UI
💬 SOP 问答
→ 聊天 UI
🔍 Collateral 筛查
→ 检索+提取卡片
📤 协议提取
→ 上传+表单+Excel下载
每个任务有自己的 UI 和后端 pipeline，共享底座（语料/权限/审计）

C · 结构化工作台 ⭐

任务是一等公民。不同任务有不同 UI（聊天/表单/文件交付/进度条）和不同后端 pipeline，但共享语料、权限、审计、追踪底座。

我的初判：C · 理由：你提到"协议提取产出 Excel"和"复杂指令/问题排查"，这两类天然不是聊天形态。A/B 都得"硬塞"。但 C 的代价是要新增"任务"这一层抽象。
