复杂指令 + 问题排查 — 五分法另两类的工程骨架

v1 不实现这两类 Agent,但骨架要清晰、接口要预留 — 否则 v2 接入时会"急造"。
这两张架构图回答:这两类 Agent 的工作流到底长什么样,和已实现的两类有什么本质差别。

① ⚙ 复杂指令执行(ToolOrchestrationAgent · v1 预留)

本质:工具链由 LLM 在运行时规划,而不是预先写死。
典型例子:用户说"给我弄张优惠券" → AI 自动完成"身份验证 → 查询用户状态 → 寻找可用券模板 → 调用发券接口 → 返回结果"等一系列精密操作。

复杂指令的关键特征

特征含义
工具链来源运行时由 LLM 规划(过滤引擎从工具库挑出本次该用的工具子集)
工具池开放 + MCP 动态接入 + 冲突检测(治理流水线)
任务终态副作用(发了券 / 改了单 / 发了邮件)
失败成本高(误调工具有真实业务后果)
Memory 用法驱动决策(个人偏好 + 项目上下文影响选哪个工具)
核心算法"过滤引擎" — 在工具库里挑出本次该用的工具子集

② 🔍 问题排查(TroubleshootAgent · v1 预留)

本质:从"表象"到"根因"的 RCA(Root Cause Analysis)过程,工具调用与人工经验融合的层层推理。
典型例子:用户说"软件用不了" → AI 像侦探一样追问、查证 → "什么时候用的? 什么错误提示? 我查一下你的账号状态... 哦,发现是账号被风控了"。

③ 三类(知识问答 / 复杂指令 / 问题排查)的本质对比

维度 💬 知识问答 ⚙ 复杂指令 🔍 问题排查
工具链 固定(检索→生成) 运行时 LLM 规划 ReAct 边走边看
工具池 不需要 开放 + MCP 动态 有限 + 诊断专用
知识 vs 工具 仅 KB 主要工具池 双源消费(KB + 工具)
任务终态 信息产出 副作用 诊断结论 + 补救
失败成本 低 高 中
Memory 用法 注入 prompt 增强 驱动决策 影响推理路径
核心算法 检索 + 重排 过滤引擎 ReAct + 状态机
关键洞察:三类 Agent 在推理形态、工具消费方式、Memory 用法、失败模型四个维度完全不同。把它们硬塞到同一个 Agent 实现里会让护栏体系撕裂。

所以问题排查不是复杂指令的子模式 — 它有自己的核心算法(ReAct + 双源消费),独立成类。
五分法的正交性(C1)在这两类的对比中得到再次验证。

④ 这两张图给底座的横切贡献 — 不是某一类专属

从复杂指令图提取的横切

  • 工具治理流水线(作者 → 冲突检测 → 工具解析 → MCP 工具池)— v2 引入
  • 评测闭环(good/bad case → 评测构造 → 反馈库)— 与 b2 评测自我迭代统一
  • Memory 双层(个人 mem + 项目 mem)— 与 v1 三件事统一

从问题排查图提取的横切

  • 管理员可配置策略层(问题分类规则 + 推理策略 / 人工经验 + 话术映射)— 给所有 Agent 用
  • 意图路由(分流到不同 Agent)— Task 编排层的能力
  • 双源消费 ReAct(KB + 工具交错)— 是工具编排的特殊形态

"工具治理 / 评测闭环 / 管理员策略层 / 意图路由"全部下沉到底座作为横切能力 — 不是某一类 Agent 专属,而是整个智能工厂的"产线公共设施"。这给了底座清晰的扩展骨架。

⑤ v1 必须为这两类 Agent 留的接口

预留位v1 落地形式v2 接入时
Agent 接口可扩展 AgentRegistry 设计为可注册任意实现 Agent 接口的类 + skill 路由机制 注册 ToolOrchestrationAgent / TroubleshootAgent + 注册新 skill,不改接口
Ledger 多 payload_type v1 注册 4 种 payload schema(含 cmd_exec / troubleshoot 占位) 新 payload 字段写入 → 不改 t_task_ledger 表
意图路由钩子 workbench TaskExecutionOrchestrator 预留分流入口 追加问题排查路由分支
Memory 注入钩子 Agent.enrichPrompt(ctx) hook · 双层 memory 已就位 项目 mem 接入(v2 范畴)
评测信号筛选器 SignalFilter 按 payload_type 路由分类规则 新增工具调用失败 / 诊断错误的 bad case 信号规则
v1 不实现 = v1 不写实际代码,但 v1 必须把"接口扩展位、表 schema 占位、Orchestrator 路由钩子、Memory 双层 hook、SignalFilter 路由"全部留好。
这是分类法的真正价值 — 给未来留好接口位,避免每次新场景都"急造"。
