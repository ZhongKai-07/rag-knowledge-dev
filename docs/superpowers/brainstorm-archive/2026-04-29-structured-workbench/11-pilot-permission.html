试点权限 — 90% 可省,但 10% 不能漏

Ops 单部门试点 = 封闭场景。这改变了权限设计的核心矛盾 — 不再是"防跨部门泄漏",而是"用户体验是否 OK + 数据是否能学"。
试点期 90% 的传统权限设计可以延后,但 User Memory 私有 / Ledger 双投影 / 文件归属 / 双级 admin 雏形 — 这 10% 不能漏。

① 试点改变了权限的"核心矛盾"

多部门正式上线场景
主要威胁:跨部门数据泄漏
  • 核心权限问题:谁能看谁的数据
  • 必须多级 admin(global / dept / kb / agent)
  • 工具调用必须分级
  • Memory 个人 vs 部门必须严格隔离
  • Ledger 必须三层可见性切片
  • 文件归属必须严格
  • ……
→
Ops 单部门试点
主要关切:UX OK 吗 / 数据能学吗
  • 跨部门数据泄漏问题:几乎不存在
  • 管理员粒度:全局 + 部门一级足够
  • 工具授权:部门内默认全开
  • Memory 隔离:依然必须严格(信任契约)
  • Ledger 可见性:一层(部门)够用
  • 文件归属:打标即可,不严校验
试点权限设计逻辑:90% 的传统权限设计可以延后,但 10% 不能延后 — 这 10% 不是"为了安全",是为了让 self-improving 学到正确的东西 + 守住用户信任契约。

③ G7 是不能省的核心 — 信任契约

关键决策(不可妥协):User Memory 严格用户私有,即使 global_admin 也不能直接看。

这不是因为权限模型限制 — 是产品的用户信任契约。员工如果发现"我的个人偏好被部门同事看到了" / "我的角色信息被领导看到了",对 AI 信任会崩。
和部门数量无关。试点期单部门也不能省。
决策点v1 试点决定理由
User Memory 写入者 仅用户自己 v1 不引入 LLM 自动抽取,避免 PII 风险
User Memory 读取者 仅用户 + Agent prompt 注入 admin 绝对不可读;Agent 注入是写入 prompt 不是读出来
用户能否导出 必须可导出 合规底线 + 信任契约
用户能否删除 hard delete(物理删除) 合规要求"被删除即不可恢复"
Insight 看 bad case 下钻 显示部门匿名 ID,不显示真实 user_id;每次下钻打 audit log 员工知道 bad case 可能被看,但何时被看是透明的(可在自己 memory 页查"我哪条对话被 admin 下钻过")

④ 试点期权限极简方案

项目试点决定
角色模型 现状不动(user / role / dept)。dept 设为 "ops",dept_admin = Ops Lead。不引入 KB / Agent admin。
双级 admin global_admin(技术部门) + dept_admin(业务部门 Lead)。视角完全分割,不共享数据视图。
工具授权 部门内默认全开。仅打标 tool.scope = "dept_internal"(为 v2 工具治理留位)。
KB / RBAC / security_level 现状链路完全沿用,零改动。AuthzPostProcessor / MetadataFilterBuilder / KbAccessCalculator 不动。
文件归属 必填 uploader_user_id + owner_dept_id。试点期 dept 内任意成员可访问(不严校验)。
Ledger 可见性 落账时 visibility_scope 字段建好(默认 DEPT),但读取路径试点期只做 own / dept 两 view,不做 global 聚合。
User Memory 严格用户私有(user_id PK) + 用户可见 / 编辑 / 删除 / 导出(合规底线)。
Dept Memory 部门共享(dept_id PK) + 仅 dept_admin 可写 + 部门全员可读。
Dept Insight v1 雏形:dept_admin 看部门聚合。bad case 下钻显示匿名 ID,打 audit log。
跨用户读取 试点期默认禁止(即使在 dept 内,user A 也不能看 user B 的对话历史)。Insight 只看脱敏聚合。
方案改动量:基本是"现状权限链路 + Memory 三件套 + Ledger 一个字段 + 文件三个字段"。试点能很快上线 — 没有"重做权限系统"的工作量。

⑤ 双级 admin 的视角分割(关键)

维度 global_admin(技术部门) dept_admin(业务部门 Lead)
角色画像 系统医生 — 平台运营者 业务管家 — 业务运营者
关心什么 系统健康 / 成本 / 链路追踪 / 异常 部门工作效率 / 知识沉淀 / AI 是否真帮上忙
Ledger 投影 技术投影:trace_id / cost_tokens / latency / status / model_name 业务投影:task_template / kb_ids / aggregated_topics / bad_case_count
看板 /global/system · /global/cost · /global/trace /admin/insight · /admin/dept-memory · /admin/proposal-inbox
不能看 不读 query 原文 / 不看 dept memory 内容 / 不下钻 bad case 不关心 token 成本 / 不看 trace 调用链 / 不看模型版本
关键洞察:两级 admin 看的不是同一份数据的不同切片,是两套完全不同的数据维度。
技术部门看 AI 平台是"系统",业务部门看 AI 平台是"工具",两个视角互不打扰。这种分离让"权限隔离"和"职责清晰"在同一个动作里完成。

⑥ 扩到第二个部门时必须补什么

v1 试点验证后扩部门时必须补:

补什么v1 是否留好钩子扩展时改动
G6 Ledger 三层可见性的全套读路径 ✅ visibility_scope 字段已建 加 dept_agg / global_agg 两个 view,无需迁移数据
G5 dept_admin / global_admin 切开的角色判定 ✅ 已是 role 类型分级 RoleService 加判定逻辑,前端按角色过滤入口
跨部门 KB 访问的精确控制 ✅ KbAccessCalculator 现状沿用 无新代码,配置即可
提议 inbox 路由(每部门 admin 看自己部门) ✅ target_dept_id 字段已建 查询 where 过滤即可

v1 留好的钩子让扩部门是配置改动 + 加 view,不是数据迁移 / schema 变更。这是分阶段切分的真正价值。

⑦ 一句话总结

试点权限的设计哲学:不重(90% 传统权限可省)、不漏(信任契约 + 必要钩子不省)。
4 个 v1 必做增量(双级 admin / Memory 双层 / Ledger 字段 / 文件归属) 落地工作量比"重做权限系统"小一个数量级,但已经为 v2 / v3 扩展铺好了所有钩子。
关键设计决策不在"做多少",在"哪些不能漏"。
