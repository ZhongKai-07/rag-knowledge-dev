"任务" vs "Agent+Skill" — 不是二选一，是两层

用户视角是任务，系统视角是能力。两个维度独立存在，把它们混到一起会出问题。

为什么"任务"必须在用户层

  • Ops 同事不是 agent 配置者——他们是任务执行者。他们的心智是"我要做 X"，不是"我要切到 Y agent，再调 Z skill"。
  • 任务 = 入口 + UI 模板 + 产出物 + 历史。聊天形态、上传+下载形态、表单形态、流程进度形态，必须由任务承载，agent 抽象承载不了 UI 差异。
  • 跨域任务的归宿问题：Collateral 筛查既检索又抽取——硬塞到"知识问答"或"文档智能"都别扭，但作为一个独立任务、内部组合两个 agent，就很自然。
  • 任务有"实例 / 历史 / 状态机"——上传一份协议、提取中、提取完、Excel 已下载。Agent 没有这种生命周期。

为什么"Agent+Skill"应该在系统层

  • 它是实现复用的载体。COB 和 SOP 任务底层共享同一个"知识问答 Agent"，区别只是绑定的语料和 prompt。
  • Agent 是有边界的能力包：哪些 skill 可调、prompt 怎么写、用哪个模型、怎么编排——全部封装。
  • 你提到的"复杂指令类、问题排查类"未来场景，本质就是再加一个 Agent（工具编排 Agent / 诊断 Agent），新任务再"组装"现有 Agent 即可。
  • 系统层 agent 化也方便接 MCP / Anthropic Agent SDK / 自研工具调用——这是技术选型自由度。
反例：如果只用 Agent+Skill，没有任务层会怎样？
· 用户进来看到 4 个 agent，点"文档智能 agent" → 它问你"想用哪个 skill？" → 选"协议提取" → 才让你上传文件。多一层无意义跳转。
· 协议提取的"上传+表单+Excel下载"UI，硬塞进"和 agent 对话"的形态会很别扭。
· "Collateral 筛查"跨两类，归不进任何 agent。
· 任务历史 / 进度 / 产出物没地方挂。
结论：你提的"文档智能 agent + skill" 是对的——但它属于系统层。用户层仍然是"任务卡片"，任务定义里写"我用文档智能 agent 的提取 skill"。两层都保留，各司其职。
