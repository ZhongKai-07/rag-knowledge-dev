协议查询的 7 个关键决策

决策点决定
主输入框完全自由文本,不强制结构 — 保留自然语言入口
引导填空给两类用户兼容:熟悉的人直接打字,不熟悉的人用下拉拼一句(主路径自由 + 兜底结构化)
实体识别透明提交后显示"已识别:X + Y · Z · W",让用户知道系统理解对了没
Counterparty 模糊匹配"华泰" 应该能命中 "Hua Tai Securities Co., Ltd." — 底层用 alias 表 + 模糊相似度
多对手方 / 多协议"华泰和 HSBC 的 ISDA&CSA" → 4 元组 counterparty/agreement 都是数组;命中文件可能多于 1 份
识别失败时不降级到向量检索 — 直接说"未找到"+ 重新填空 + 可选"附加:试试模糊检索 →"
追问入口字段卡下方"再问 X 的另一个字段" — 保留 Counterparty + Agreement,字段位空出

协议提取的 8 个关键决策

决策点决定
上传形态拖拽 + 点击两可
不让用户选字段字段集是协议类型决定的(ISDA / CSA / GMRA / ISDA+CSA 各有固定字段集)
metadata 自动识别上传后展示"已识别:Counterparty / Agreement / 日期",一次确认
进度形态进度条 + 当前阶段(解析 / 抽取 / 校验 / 渲染) — 不显示技术细节如"调用 LLM 中"
字段卡置信度三色:🟢 ≥0.85 / 🟡 0.6-0.85 / 🔴 <0.6 或 N/A
低置信字段交互🟡 显示候选值 + [确认/改] · 🔴 [跳过/手填]
同步 vs 异步<30s 同步页面停留 · ≥30s 切异步可关页面 + 完成时通知
用户改字段后不自动重抽 · 仅更新本字段 · 标 human_corrected = true 进 Ledger
human_corrected = true 这件事很重要 — 它是"用户行为反哺评测"在 UX 上的物理实现。
用户改一个字段就是给系统一个明确的 bad signal,不需要 thumbs-down 也能进 b2 评测自我迭代闭环。这在 v1 没有 thumbs UI 的情况下,是额外的 30% bad case 信号源。

④ 字段卡 ↔ 来源卡 — 统一三段式

元素来源卡(知识问答)字段卡(文档智能)
主体 文档名 + 章节 + 片段预览 字段名 + 抽出值
置信度 不显示(v1) 三色 + 数值
跳转 → 原文(KB 文档 viewer) → 原文(PDF 第 N 页 + 高亮)
视觉容器 浅灰边框卡片,左侧 📎 浅灰边框卡片,无图标
状态色 中性灰 🟢🟡🔴
统一三段式:主信息 → 信息源 → 跳转回原文。
用户在不同 Agent / 不同 skill 之间切换时没有学习成本 — 所有 AI 产出都"长得一样"。
这是"可追溯不变量"(I4)在视觉层的物理体现 — 用户视觉上 internalize"可追溯"是产品的硬承诺。

⑤ 文档处理的诚实失败 6 种

失败种类UX 形态
实体识别失败(查询) "无法识别 Counterparty / Agreement / 字段,请重新填空 → [▾ Counterparty] [▾ Agreement] [+ 字段]" + 可选"附加:试试模糊检索 →"
未匹配协议(查询) "未找到该 Counterparty + Agreement 的协议。可能 (1) 该协议未归档 (2) 对手方简称不准 [📨 告诉 Lead 缺资料]"
文件解析失败(扫描件) "这是扫描件,需要先 OCR。当前不支持自动 OCR — [联系 IT 处理]"
metadata 识别失败(提取) "无法识别协议类型,请手动选择 → [ISDA / CSA / GMRA / ISDA+CSA]"
字段全空(提取) "解析了文档但没找到任何 ISDA 字段。可能这不是 ISDA → [切换 schema] / [联系支持]"
超时(>30s) "处理时间较长,已切到后台。完成时会通知你。可以关闭这个页面。"
