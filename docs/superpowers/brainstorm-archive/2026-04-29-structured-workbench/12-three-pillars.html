持续学习层的核心三件事

试点期不需要重权限管理,但 User Memory / Dept Memory / Dept Insight 必须做对。
它们不是三个独立功能 — 是同一个 self-improving 闭环上的三个支点。

① 三件事各是什么

个人层 · 私有

👤 User Memory

用户主动告诉 AI"关于我,请记住这些"的事实集合。注入个人 prompt。

  • 角色 / 偏好 / 关注主题 / 习惯术语 / 个人快捷
  • 用户随时可看 / 可编辑 / 可删除 / 可导出
  • 严格私有 — 即使 admin 也不能直接看
  • v1 上限 50 条
部门层 · 共享

🏢 Dept Memory

部门内对所有人 Agent 行为生效的"部门级共识"。注入所有部门成员的 prompt。

  • 术语 / 规则 / 风格 / 警示 / 流程
  • 由 dept admin 维护
  • 部门全员可读,仅 admin 可写
  • v1 上限 100 条
运营层 · 聚合

📊 Dept Insight

Ledger 数据按部门聚合后的脱敏可视化。给 dept admin 决策"AI 在我部门用得怎么样"。

  • 任务用量 / 高频主题 / bad case / 命中率
  • 仅 dept admin 可见
  • 默认聚合不暴露个人,bad case 下钻打 audit log
  • self-improving 提议机制的素材库

② 三件事的关系 — 闭环图

🗃 底座 Ledger(所有任务执行结构化落账,部门可见)
↓ 双投影 ↓
👤 User Memory 用户主动写
🏢 Dept Memory admin 维护
📊 Dept Insight 系统聚合给 admin
↓ 注入 / 反哺 ↓
下次 Agent 执行 注入两层 memory · 受益于 admin 改进
↑ 自下而上(promote) 用户在 User Memory 写了一条偏部门通用的事实 → 主动提议提升 → admin 在 inbox approve → 复制(不是迁移)到 Dept Memory。提议人那条保留,标记"已提升到部门"。
↓ 自上而下(insight 驱动改进) admin 看 Insight 看板发现 bad case 集中 / 高频主题 / 死规则 → 直接改 Dept Memory / 派发 KB 补充工单 / 下线无效规则。
⟲ 系统主动提议 规则引擎扫 Ledger → 生成 4 类提议放进 admin inbox → admin 一键 Approve / Reject → 写回 Dept Memory / KB 工单。
核心闭环:Ledger 是燃料 · Insight 是 admin 的眼睛 · Dept Memory 是部门的记忆 + Insight 改进的落点 · User Memory 是个人的记忆 + Dept Memory 的人才储备 · promote + insight + 系统提议 三条通道让闭环跑起来。

③ 产品约束(信任契约)

关键决策User MemoryDept MemoryDept Insight
谁可写 用户自己 仅 dept admin 系统(只读)
谁可读 仅用户自己 + Agent 注入 部门全员 + Agent 注入 仅 dept admin
admin 能看吗 绝对不能 是,自己维护 是,主用
个人 ID 暴露 — — 聚合默认匿名;bad case 下钻显示部门匿名 ID
下钻打 audit log — — 每次下钻必打;用户在自己 memory 页可看"我哪条对话被 admin 下钻过"
导出 用户可导出自己的 不导出 不导出(防止数据外流)
删除是否可逆 hard delete(合规底线) 软删 30 天可恢复 —
关键边界:User Memory 严格私有 — 这不是因为权限模型限制,而是用户信任契约。员工如果发现"我的个人偏好被部门同事看到了",对 AI 信任会崩。和部门数量无关,试点期也不能省。

④ promote 流程 — 个人沉淀部门

流程示意
1. 用户在 /profile/memory 写下: "EUM = EU MTA"
↓ 点击 [提议给 Ops Lead] 按钮
2. admin 看板 /admin/memory-inbox 出现一项:
提议人: 张三 · 类型: 术语 · 内容: "EUM = EU MTA"
↓
3a. admin Approve → 复制到 t_dept_memory(标记 source_user_id) · 用户该条标记"已提升到部门"
3b. admin Reject → 用户收到通知 + 原因
"复制式提升"的设计原因:用户 User Memory 那条保留,只是状态变成"已提升到部门"。这样 — 用户离职/调岗时个人 memory 删除不影响 dept memory;dept admin 修改/下线 dept memory 不影响用户那条。所有权清楚,不互相阻塞。

⑤ Memory 注入到 Agent prompt 的具体形态

每次 Agent.execute 的 prompt 构造
[基础 system prompt]
+ [部门上下文]   ← 注入 Dept Memory(去重后的核心规则/术语)
+ [个人上下文]   ← 注入 User Memory(按 type 分组渲染)
+ [用户当前 query]

注入次序固定:Dept 先于 User — 部门规则是约束(不能违背),个人偏好是偏好(在约束下生效)。两者冲突时部门胜出。

token 预算:User Memory 注入段 ≤ 500 tokens(超出按相关性截断,v1 用 query 词袋粗判)。

用户面 Memory 透明栏(屏 2a 知识问答)

默认折叠 → 单行小灰字
ⓘ 基于你的角色(EU Ops collateral analyst)+1 部门规则     [展开]

展开后显示具体注入了什么 + "编辑我的记忆 →" 快捷入口直接进 /profile/memory。让"AI 越用越懂我"有反馈回路。

⑥ 数据库 schema 草案(v1 试点)

表核心字段用途
t_user_memory user_id, type, content, tags, promoted_to_dept_id 个人记忆 · type ∈ {role, preference, topic, term, shortcut}
t_dept_memory dept_id, type, content, priority, source_user_id, created_by 部门记忆 · type ∈ {term, rule, style, warning, process}
t_memory_promote_request user_memory_id, user_id, dept_id, status, reviewed_by, review_note promote 提议箱(用户 → 部门)
t_memory_injection_log task_instance_id, memory_type, memory_id 注入命中日志 · 用于 Dept Memory 命中率统计
t_insight_audit_log admin_id, action, target_ledger_id bad case 下钻审计 · 用户可看"我哪条被看过"

共 5 张新表;实际改动量比"重做权限系统"小一个数量级。详细 SQL 见 spec(待 M1-M13 修正后回写)。

能力v1 试点v2 完整
