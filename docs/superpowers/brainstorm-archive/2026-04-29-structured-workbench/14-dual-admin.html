双级 admin — 看完全不同的世界

技术部门和业务部门看 AI 平台,该看的事完全不同 — 不是"同一份数据的不同切片",是两套完全不同的数据维度。
把这两个角色的视角彻底分开,让权限隔离和职责清晰在同一个动作里完成。

① 两个角色的产品画像

🩺
技术部门

global_admin · 系统医生

平台运营者 · 关心 AI 平台是不是健康的"系统"。
每天想知道:今天 AI 跑得稳吗 / 烧了多少钱 / 哪条链路出错了。
看什么
系统健康(错误率 / p95 延迟 / 模型可用性 / SLA)
成本(总 token / 各模型用量 / 部门分布)
链路追踪(按 trace_id 查全链路 span 树)
不看什么
业务 query 原文 / 答案内容
Dept Memory 内容
User Memory(绝对不可看)
bad case 业务下钻
典型问题
  • "今天哪个模型挂了?"
  • "上月 token 烧了多少?"
  • "trace_id=abc 这次问答全链路是怎么走的?"
📋
业务部门

dept_admin · 业务管家

业务运营者(部门 Lead)· 关心 AI 是不是真在帮部门干活。
每天想知道:部门用 AI 干什么多 / 哪里 AI 答得不好 / 系统给我什么提议。
看什么
Dept Insight(任务用量 / 高频主题 / bad case)
Dept Memory(部门规则 / 注入命中率)
系统提议 inbox(沉淀部门记忆 / KB 补充 / 死条清理)
用户提议 inbox(promote 审批)
不看什么
token 成本 / 模型版本
trace 调用链
系统错误率 / SLA
User Memory(绝对不可看)
个人 user_id(看 bad case 走匿名 ID)
典型问题
  • "我部门最近问 AI 什么多?"
  • "哪类问题 AI 答得不好?"
  • "系统建议我做什么?"

② Ledger 双投影 — 同一份数据,两个完全独立的读取通道

🗃 底座 Ledger(每次任务执行结构化落账)
↓ 技术投影 ↓
↓ 业务投影 ↓
给 global_admin
字段:trace_id / cost_tokens / cost_usd / duration_ms / status / model_name / latency_p95
→ 技术看板(系统健康 / 成本 / 链路)
给 dept_admin
字段:task_template / kb_ids / aggregated_topics / bad_case_count / memory_hit_rate
→ 业务看板(Dept Insight / 提议 inbox)
关键约束:这两个投影永远不混合。一份 Ledger 表,两个完全独立的读取通道。

不是"权限隔离"得到的结果,是"产品视角分割"的物理实现。dept_admin 的看板上找不到任何 token / trace 字段;global_admin 的看板上找不到任何 query 内容。
在 v1 就要做对,否则 v2 时要打散补 — 数据投影分裂不是事后能切干净的事。

两个看板放在一起 — 一个像"运维仪表盘",一个像"业务管理后台"。完全不同的视觉语言、不同的关注维度,各自满足角色心智。

④ 双级 admin 设计的 4 条原则

#原则含义
PA1 admin 路径也禁技术词 dept_admin 是业务管家不是工程师 · 看板里禁 "Agent / Skill / Task 模板 / Prompt / Schema" 字眼
PA2 默认脱敏 + 聚合 admin 看到的所有数据默认脱敏 · 下钻到原文必须打 audit log · 用户可看"我哪条被下钻过"
PA3 核心动作是一键决策 admin 的核心动作是 Approve / Reject / Mark / Drilldown · 不是复杂配置
PA4 双级看板完全独立 同一组件树、不同投影 · 不混合视图 · 不共享数据维度

⑤ v1 / v2 / v3 演进

阶段admin 层级具体能力
v1 试点 双级:global_admin + dept_admin 视角完全分割 · 共 8 屏(global 3 屏 + dept 5 屏) · 双投影 Ledger
v2 扩部门 + kb_admin + agent_admin 跨部门 KB 共享精细化 · 工具治理多级 · Autodream 写 Memory 时打 audit
v3 全员 + OPA / OpenFGA(灵活授权) 跨域授权 · 反哺 KB 审核流

v1 双级雏形已经为 v2 / v3 扩展铺好接口。每一阶段是"加新角色 + 改 RoleService 判定逻辑",不破坏 v1 的数据视图。

⑥ 实现细节

角色判定服务

RoleService:
  isGlobalAdmin(user) → user.roles contains GLOBAL_ADMIN
  isDeptAdmin(user, dept_id) → user.roles contains DEPT_ADMIN AND user.dept_id == dept_id
  canViewSystemDashboard(user) → isGlobalAdmin(user)
  canViewDeptInsight(user, dept_id) → isDeptAdmin(user, dept_id) OR isGlobalAdmin(user)
  canDrilldownBadCase(user, ledger_row) → isDeptAdmin(user, ledger_row.dept_id)
  canReadUserMemory(user, memory.user_id) → user.id == memory.user_id  // 仅本人

前端路由分割

路由守卫对应屏
/global/*RequireGlobalAdmin屏 8 / 9 / 10
/admin/*RequireDeptAdmin(dept_id)屏 3 / 4 / 5 / 6 / 7
/profile/memoryRequireAuth个人 memory 页
/workbenchRequireAuth主工作台(屏 1 / 2a / 2b)

导航栏行为

  • 普通用户:看到 /workbench + /profile/memory
  • dept_admin:额外看到 /admin/* 入口
  • global_admin:额外看到 /global/* 入口
  • 三种入口在同一个顶栏切换 — 用户切换时不跳出当前会话(沿用现有 SPA 行为)

⑦ 一句话总结

双级 admin = 两个完全不同的视角 · 同一份 Ledger 双投影。
技术部门是系统医生 — 看健康 / 成本 / 链路;业务部门是业务管家 — 看 Insight / Memory / 提议。
两个看板互不打扰 — 这种分离让"权限隔离"和"职责清晰"在同一个动作里完成,也是"避免向业务暴露技术语言"在 admin 路径的物理保证。
