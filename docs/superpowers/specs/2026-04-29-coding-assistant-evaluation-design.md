## 1. 背景与决策目标

### 1.1 背景

公司内部各岗位（研发、量化、算法、EST、PWM、业务、产品、运营）对 AI 代码助手的依赖快速增长。当前缺少集中管控的工具选型，存在三类问题：

- **合规风险**：核心资产（量化策略、EST 客户/订单数据）通过个人版工具流向境外厂商，无审计无追溯。
- **场景错配**：研发部门的工具不一定适合业务/PoC 场景，反之亦然，统一上一款会牺牲适配。
- **采购分散**：缺统一名单和 license 池，成本与治理不可控。

### 1.2 决策目标

输出一份"**部门 × 工具适配方案**"，支撑技术管理层和产品经理做出"**B 方案：按部门组合 2~3 款工具**"的采购决策，并附带：

- 推荐组合 + 落地路线图
- 风险红线 + 退出策略
- 合规约束清单（部分由法务/合规进一步核对）

### 1.3 非目标（明确不做的事）

- 不做工具的"统一打分排名"。每款工具有其最佳场景，不评"哪款最好"。
- 不做完整 TCO 财务分析。粗估单座 license 量级即可，详细采购另谈。
- 不做监管合规深度分析（证监会/交易所细则、等保 2.0、《数据出境安全评估办法》等），仅在合规章节列开放清单提示读者去找法务。
- 不评测 Cursor / Windsurf / JetBrains AI / 通义灵码等 — 受企业规划、采购难度、地区限制，三款候选已锁定。

## 2. 候选工具与基准

### 2.1 深度评测对象（3 款）

| 工具 | 厂商 | 形态 |
|---|---|---|
| **GitHub Copilot Enterprise** | GitHub / Microsoft | IDE 插件 + Web + CLI（gh copilot） |
| **Gemini Code Assist (Enterprise / Standard)** | Google Cloud | IDE 插件 + Gemini CLI |
| **AWS Kiro** | AWS | 独立 IDE（fork VS Code）+ spec-driven workflow |

注意：Gemini Code Assist 的 **Enterprise 与 Standard 双版本**会做对比，因为差异点不在模型本身而在企业治理 / 上下文 / 私有代码库索引。

### 2.2 基准对照（不深评，仅在关键能力点点名）

| 工具 | 用作基准的能力点 |
|---|---|
| **Claude Code** | Skills / MCP / Subagents / Plan mode / Memory / 探索式上下文理解 |
| **Codex（CLI + Cloud）** | Agentic 任务完成率天花板 |

基准对照只出现在 § 3.1 / § 3.3 / § 3.4 / § 3.6 四处，不展开成完整章节。

## 3. 部门画像与场景（共 8 个部门）

公司行业语境：**证券金融公司**（不是互联网公司）。这影响 EST、PWM、量化等部门的画像与合规约束。

| 部门 | 典型角色 | 核心场景 | 关键诉求 |
|---|---|---|---|
| **研发**（含 CI/CD 子场景） | 后端/前端/全栈/SRE | 大型复杂项目长期开发、跨模块重构、Spec-driven design、CR、bug 修复、CI/CD 流水线 | 仓库级上下文、多文件 agent、plan mode、稳定 IDE、长生命周期 memory、Git/PR 集成 |
| **量化研究** | 量化研究员、策略开发 | 因子开发、回测框架、策略代码、信号挖掘（Python/C++/KDB+） | 数学/统计正确性、Notebook 友好、回测代码模式识别、**严格数据隔离（策略/因子是核心资产）**、可能需 BYOK |
| **算法（AI）** | ML/DL 工程师 | 模型训练/推理、特征工程、PyTorch/HuggingFace 生态 | 长上下文（论文级 prompt）、ML 框架熟悉度、Notebook + 训练脚本、GPU/集群脚本、调参实验追踪 |
| **EST**（Equities Sales & Trading） | 交易员、销售交易、交易支持 | 盘中数据查询、一次性 Python/VBA/SQL、Bloomberg/Wind 集成、Excel 自动化 | **极低门槛、自然语言强、Web/桌面优先、不要求 Git、强合规（订单/客户敏感）** |
| **PWM 产品数据分析** | PWM 产品经理、客户数据分析 | 客户画像、产品业绩、SQL + Excel + Pandas、报表生成 | SQL/Excel/Pandas 强、自然语言→数据查询、一次性脚本 |
| **业务通用** | BA、运营支持 | 数据拉取、报表自动化、流程小工具 | 极低门槛、模板驱动、可能完全不接触 Git |
| **产品经理 / PoC** | PM、产品设计 | PoC 原型、Demo 验证、轻量前端 | 0-1 项目生成、UI 预览、快速迭代、不要求工程化 |
| **运营** | 运营、内容、增长 | 简单脚本、报表、可能 RPA | 极低门槛、模板化 |

部门画像在主文档第 1 章会展开成"画像 + 推荐 + 不推荐 + 风险红线 + 小贴士"五栏。

## 4. 评测维度框架

### 4.1 维度分组与详略策略

按"决策性 vs 背景性"分两类，详写 3 类、轻写 3 类。

#### 详写（决策性维度）

- **A. Harness Engineering**（agent 框架工程）
  系统提示与上下文构造 / 工具调用机制 / Plan mode / Spec-driven 流程 / Subagent 编排 / Memory（会话内 / 跨会话 / 项目级）/ 权限模型
  ★ 基准对照：Claude Code 的 plan mode + subagents + skills + TodoWrite + permission tiers

- **D. 代码理解与上下文深度**
  单仓索引规模 / 增量索引时延 / 跨仓上下文 / 代码图（call graph / 依赖图）感知 / 长上下文实际可用率（声称 vs 实际）
  ★ 基准对照：Claude Code 的 grep + agent 探索式理解 vs 索引式理解的差异

- **F. 扩展性与生态**
  IDE/CLI 覆盖度 / **MCP 支持现状（重点）** / Plugin / Extension / Skills 类机制 / 与公司既有系统集成（Jira / Confluence / 内部知识库 / 私有 git）
  ★ 基准对照：Claude Code 的 MCP + plugins + skills 三层扩展模型

#### 轻写（背景性维度）

- **B. 模型支持与路由** — 一张表搞定（默认模型 / 可选 / 切换粒度 / BYOK / 更新节奏）
- **C. 模型 coding/agentic 能力** — 引用 SWE-bench Verified / Terminal-bench / LiveCodeBench 等公开 benchmark + 一段实战体感
- **E. 可靠性与企业治理** — SLA / SSO / SCIM / RBAC / 审计；与第 7 章合规清单交叉引用避免重复

### 4.2 评分口径

不打数字分。采用三档定性评级：**强 / 一般 / 弱**，每条都附"理由 + 关键论据来源"。理由：

- 数字分容易被各部门拿去抠
- 三家企业版的真实差异更适合质性表述
- 评测时没有完全控制变量的实测条件，伪精确不诚实

## 5. 合规与可用性约束（瘦身版）

主文档第 7 章只写以下 3 条 + 1 个开放清单，不做监管深度分析。

### 5.1 硬约束 1 — 大陆可用性

| 工具 | 大陆企业版可用性 | 备注 |
|---|---|---|
| GitHub Copilot Enterprise | 走代理可用，无大陆区，流量出境到 GitHub/Azure | 个人版国内已正常使用，企业版同链路 |
| Gemini Code Assist Enterprise / Standard | 走代理可用，无大陆区，流量出境到 Google Cloud | 两版同 |
| AWS Kiro | 走代理可用，**截至 2026-04 仍在 Preview 且区域有限**，无北京/宁夏区 | 三家里成熟度最低 |

结论：三款都是"出境 + 走代理"路线，**任何部门要用都默认接受 prompt 出境到海外厂商**。

### 5.2 硬约束 2 — 数据训练 / 留存

各家企业版的"零训练 / 零留存 / CMEK"承诺对照表，引用条款链接。写文档时去查最新版（条款会变）。

### 5.3 说明 3 — 核心资产泄露红线

量化策略代码、EST 客户/订单数据 = 核心资产。三款企业版"零训练" ≠ "零风险"：

- Prompt 仍上传厂商服务器做推理（即使不训练）
- 一旦服务商被攻破 / 内鬼 / 司法调取，数据可能落入第三方

**红线规则**：核心资产相关的代码 / 数据建议规避走 SaaS 路径。这条在第 1 章每个相关部门的"风险红线"一栏复述。

### 5.4 开放清单（写在主文档第 7 章末尾，提示读者去找合规/法务核对）

- 跨境数据传输的具体监管要求（《数据出境安全评估办法》《个人信息保护法》）
- 证监会 / 交易所对 AI 工具的审计要求
- 等保 2.0 / 数据分级
- 出问题的责任界定 / 合同条款细节
- 各家是否提供 SOC2 / ISO27001 报告

   3.3 模型 coding/agentic 能力      [轻 + Codex 基准]

### 6.2 各章核心内容要点

#### 第 0 章 TL;DR
- 一句话结论
- 部门 × 工具推荐矩阵图（一张大表）
- 三款工具一句话定位
- 关键风险红线

#### 第 1 章 部门画像与推荐（决策入口）
每个部门 5 栏：画像 / 首选 + 理由 / 次选 + 适用条件 / 不推荐 + 原因 / 风险红线 + 小贴士。
研发部门下设 CI/CD 子场景。

#### 第 2 章 三款工具速览
每款"能力名片"：厂商定位、产品形态、核心卖点、大陆可用性、默认/可切换模型、计费、~80 字总评。
**特别处理**：AWS Kiro 标"Preview 阶段成熟度警示"栏。

#### 第 3 章 核心能力对比
按 4.1 的详略策略写。每节固定结构：
1. 维度说明（这条为什么重要）
2. 三款现状（强 / 一般 / 弱 + 理由）
3. （详写章）★ 基准对照 — Claude Code 或 Codex 的对应能力一段
4. 对部门选型的影响（一段）

#### 第 4 章 适配矩阵
8 部门 × 3 工具 × {推荐度 / 关键理由 / 风险} 大表。视觉收束第 1 章和第 3 章。

#### 第 5 章 推荐组合（B 方案）
- 主力工具 + 备选工具组合
- 哪些部门跑哪个
- License 池粗估
- 与 Claude Code / Codex 类基准的关系（"未来评估窗口"，标注哪些场景如能用上 Claude Code 会显著提升，作为长期视角）

#### 第 6 章 风险与路线图
- 风险表（合规 / 锁定 / 成熟度 / 数据）
- 试点 → 推广 → 全量 三阶段路线图（粗略时间线，无强承诺日期）
- 退出策略

#### 第 7 章 合规约束
按 § 5 的瘦身版结构。

#### 第 8 章 附录
- benchmark 数据原始引用
- 厂商企业版条款摘录与链接
- 评测方法与口径（评分如何打、为什么不上 Excel 评分表）
- 参考资料

## 8. 评测方法与信息来源

### 8.1 信息来源优先级

1. **一手**：厂商官方文档、企业版条款、定价页、产品博客、API/SDK 文档
2. **二手**：技术分析报告、独立评测、社区 benchmark（标注发布日期，避免引用过旧数据）
3. **基准对照**：Claude Code / Codex 的对应能力，引用其官方文档 / Anthropic / OpenAI 公开材料

### 8.2 实测投入

不做完整三家并行实测（成本和时间不允许）。但允许：

- 各厂商已公开的 demo 视频 / 文档示例做能力对照
- Claude Code 自身能力可在写作过程中作为对照基准（手边即工具）
- 三款工具中如有团队成员有实际使用经验，纳入"实战体感"段落，标注"基于 X 部门同事 N 周使用反馈"

### 8.3 时效性声明

文档末尾标注"截至 2026-04-29"。三款工具均为快速迭代产品，特别是 AWS Kiro（Preview）和 Gemini Code Assist 的功能矩阵 6 个月内可能显著变化。退出策略和路线图章节会强调"半年内复审"机制。

## 9. 目标读者与产出风格

### 9.1 读者

- **技术管理层**：要快速看到结论 + 关键技术差异 + 能背书的论据
- **产品经理**：要场景适配 + 直观可视 + 案例感

第 0 章和第 1 章服务两类读者；第 3 章主要给技术管理层；第 4 章是两类读者的共同收束。

### 9.2 风格

- 中文为主，工具名/产品名/术语保留英文
- 每章开头一段 TL;DR
- 不堆砌厂商营销话术，直接说能力差异
- 对不确定的能力点诚实标注"待实测确认"或"基于厂商声称"
- 不做"哪款最好"的总分排名 — 每款都有最佳场景

## 10. 风险与已知不确定性

### 10.1 信息时效

三款工具都在快速迭代。文档落稿后 3~6 个月内可能需要修订。spec 不预先承诺修订节奏，由读者根据自身需要回查。

### 10.2 AWS Kiro 成熟度

Kiro 当前 Preview，区域和能力都在变。如果落稿前 Kiro 进入 GA 或退出预览状态变化，需在主文档第 2 章和第 5 章相应更新。

### 10.3 厂商条款变化

§ 5 的合规约束和 § 8 的附录都依赖厂商条款。条款变化会影响推荐结论。落稿前需 cross-check 一遍最新条款。

### 10.4 实测覆盖有限

未做并行实测，主要依赖文档 + benchmark + 个别使用反馈。第 3 章某些细节（如索引规模上限实测、长上下文实际可用率）可能只有"厂商声称"而无实测验证 — 这类点会显式标注。
