§ 01

三层架构总览

把整个平台想成三层:任务工作台层、任务原语层、能力原子层。每层职责清晰,层间关系决定了整个系统的演化弹性。

这个分层最关键的好处是: Layer 1 与 Layer 2 是 1:1 关系(每个工作台对应一种原语),Layer 2 与 Layer 3 是 N:M 关系(每个原语调用多个原子)。这让系统可以 "上层快速增加新工作台,下层稳定复用基础能力"——这是平台型架构的本质。

图 01三层架构 · 工作台 / 任务原语 / 能力原子

三层之间的对应关系是这套架构能够长期演化的关键: 上层增加新业务场景,只需要新建一个工作台和对应原语;下层基础能力一次建好,被所有原语共用。这避免了 "每个新场景都要从头搭一套" 的反模式。

能力原子是稳定的,任务原语是有限的,工作台是可扩展的。三者各司其职,系统才长得大。

§ 02

当前 RAG 系统 vs 目标架构 Gap

现有 6 个域(RAG / Knowledge / Ingestion / Admin / Eval / Core)对照目标架构,在 8 个维度上存在 Gap。Gap 严重度决定了重构的优先级。

# 维度 当前态 目标态 严重度
01 任务抽象 无 Task 概念,所有请求按 "问答" 处理 Task 为一等公民,有类型 / 状态 / 输入 / 产物 / 轨迹 高
02 任务路由 单一 RAG pipeline 按任务类型分发到不同 Executor 高
03 抽取能力 无 字段级 Schema 抽取 + 置信度 + Maker-Checker 高
04 UI 形态 单一 Chatbot 多形态工作台(对话流 / 表单 / 工作流) 高
05 工具调用 无 Tool Registry + 安全沙箱 + 审计 中
06 评估指标 RAGAS 类(只覆盖 QA) 按原语区分: QA 准确性 / 字段 F1 / 任务成功率 中
07 沉淀机制 假设是 Q&A 形态 按原语区分(Q&A 结晶 / 抽取模板 / Tool 模式) 低
08 权限模型 RBAC + ABAC + ACL 覆盖文档 还需覆盖任务级 + 产物级权限 中

最痛的三个 Gap 是 01、02、03——它们是阻塞性的,不解决就无法承载 Colleteral 两个新场景。其余 Gap 可以渐进补齐。

§ 03

Task 域的核心抽象

新增的 Task 域是整个重构的中枢。它定义了任务的领域模型、生命周期、执行接口、产物管理。

Task 域承担四件事: 定义 Task 实体(领域模型) · 提供 TaskExecutor 接口(供各执行器实现) · 路由 Task 到对应 Executor(TaskRouter) · 管理 Task 产物的归属与回流(ArtifactRegistry)。

图 02Task 域的核心实体与接口

这套设计的核心妙处: 现有 RAG 系统不需要推翻重建,只需要包一层 RetrievalQAExecutor 的适配器,就能纳入新架构。这是 Phase 1 能在两个月内完成的根本原因。

§ 04

各域职责划分

从现有 6 个域演化到目标的 8 个域。绿色 = 已存在可复用,黄色 = 需改造,红色 = 全新建设。

Task 域

新建
所有任务的中枢。定义 Task 实体、TaskExecutor 接口、TaskRouter、ArtifactRegistry。
  • Task 生命周期管理(创建 / 执行 / 完成 / 失败 / 归档)
  • 任务路由分发
  • 产物登记与归属
  • 统一的任务级权限控制

RAG 域

改造
从独立的问答处理域,改造为 RetrievalQAExecutor 的实现。原有 ReAct loop、RRF 等能力保留。
  • 实现 TaskExecutor 接口
  • 对外暴露为 Retrieval QA 原语
  • 检索能力沉淀到能力原子层
  • 不再直接面向用户

Extraction 域

新建
承担文档抽取与混合检索抽取。包含模板管理、字段抽取、置信度、Maker-Checker。
  • 实现 ExtractionExecutor、HybridExecutor
  • ExtractionTemplate 模板中心
  • 字段级置信度评估
  • 抽取结果回流到 Knowledge 域

Knowledge 域

复用
承担知识空间、文档资产、权限管理。现有能力延续,新增任务产物的接收能力。
  • 多 Space 多租户管理
  • RBAC + ABAC + ACL 权限
  • 新增: 任务产物注册接口
  • 新增: 沉淀的 Q&A / 字段索引

Ingestion 域

复用
文档处理编排不变。新增对 Extraction 域的支持: 解析时同步生成抽取所需的章节定位信息。
  • BDA / Tika 解析编排
  • 分块与索引
  • 新增: 章节级元数据(供抽取定位)
  • 新增: 同步入抽取专用索引

Core 域

复用
文档解析与分块策略,作为能力原子层的一部分。供 Ingestion / Extraction 共用。
  • 文档解析能力
  • 分块策略
  • 作为 Capability Atom 暴露
  • 多原语共用

Eval 域

改造
从单一 RAGAS 评估,扩展为按原语区分的评估体系。每种原语有专属指标。
  • QA: RAGAS faithfulness / relevance
  • Extraction: 字段 F1 / Recall / Precision
  • Hybrid: 精确匹配率
  • Multi-step: 任务成功率

Admin 域

改造
后台仪表盘。从 RAG 链路追踪扩展为任务级追踪 + 产物管理 + 沉淀治理。
  • 任务执行监控
  • 多原语链路追踪
  • 产物治理与回流审核
  • 合规审计与报表

§ 06

渐进重构路径

四个阶段、约十个月。先重构骨架再扩张能力,避免形成两套并行系统。

核心策略: 先重构再扩张。先把 Task 抽象和路由做出来,再叠加新能力。如果直接在当前 RAG 之上 "加" Extraction 模块,会形成两个并行系统,后期再合并的代价会非常高。

01

任务化骨架

PHASE 1 · 1—2 月 · 重构
把现有 RAG 系统纳入新架构,但对外行为不变。这是必经的"架构债偿还"阶段。
  • 新增 Task 域,定义 Task 实体与生命周期
  • 抽象 TaskExecutor 接口,实现 TaskRouter
  • 把 RAG 域重构为 RetrievalQAExecutor 适配实现
  • Chatbot 入口改为 "知识问答工作台" UI 实例
  • Task 级权限模型扩展
  • 新增 ArtifactRegistry 框架(暂不实现回流)
交付物
用户感知不变,后端架构完成升级。所有 QA 任务通过新 Task 域执行。可观测性显著提升。
02

文档抽取 + 混合筛查

PHASE 2 · 2—4 月 · 扩张
交付 Ops 最迫切的两个新场景: 协议智能提取、Colleteral 智能筛查。知识飞轮在这里第一次转起来。
  • 新增抽取引擎(LLM + Structured Output + JSON Schema)
  • 实现 ExtractionExecutor(协议解析 Pipeline)
  • 实现 HybridExecutor(检索 → 定位 → 抽取)
  • Maker-Checker 复核工作流
  • ISDA / CSA / GMRA 模板与字段 Schema 设计
  • 新建文档抽取工作台 UI
  • 新建混合筛查工作台 UI
  • 实现 Artifact 注册回 Knowledge 域
交付物
Colleteral 两个场景可上线试运行。抽取产物开始反哺筛查任务,飞轮闭环初步成立。
03

工具调用基础设施 + 复杂指令

PHASE 3 · 4—7 月 · 进阶
从被动响应走向主动执行的分水岭。合规风险最高,任何 Tool 都可能产生副作用,必须有完整的用户确认 + 审计 + 回滚机制。
  • Tool Registry + 参数验证 + 执行沙箱 + 审计日志
  • 实现 MultiStepExecutor(Plan-Act-Reflect loop)
  • 初始 Tool 集合: CloudMargin 查询、协议库查询、对比报告生成
  • 用户确认与中止机制
  • 新建复杂指令工作台 UI(规划可视化)
  • 任务模板规划
交付物
第一批受控的复杂指令可上线。所有指令都有完整审计链。
04

问题排查 + Memory 启动

PHASE 4 · 7—10 月 · 收尾
最难、最不可控的原语。准确率天然受限,定位必须严守 "AI 给的不是答案,是排查思路"。Memory 机制在此阶段启动,严格 Space 隔离。
  • 实现 DiagnosticExecutor(假设-验证循环)
  • 错误码字典与历史排查案例库
  • 新建问题排查工作台 UI
  • 用户级 Memory 机制启动(严格 Space 隔离)
  • Pulse 仅做内部观察类(不碰对客)
  • 任务模板复用功能上线
交付物
五大原语全部上线。平台进入持续运营阶段,飞轮全速运转。
§ 07

风险与对策

这套架构在落地过程中可能遇到的真实风险。提前识别,提前对冲。

高风险

Phase 1 重构 "无产出"

对内对外都看不到新功能,容易被业务方质疑 "在做什么"。这是架构债偿还期的典型困境。

对策: 在 Phase 1 末尾交付 "可观测性提升" 这一可见价值 —— 任务级追踪面板、统一指标看板。让业务方看到底层升级带来的运维和评估能力提升。
高风险

"Agent 平台" 的产品形态滑坡

团队内部讨论时,容易把 "工作台" 滑成 "创建一个个 Agent",最终做成 Coze 的小弟版,丢掉差异化。

对策: 严格执行 §08 术语统一表(见产品文档)。代码层面,Executor 类不要叫 *Agent;UI 层面,任务一定先于 Executor。每次设计评审强调 "任务先行"。
高风险

抽取准确率不达标

金融协议字段抽取,准确率必须过 95% 才有业务价值。低于此值,Maker-Checker 工作量比手工录入还大。

对策: Phase 2 开始前,先用一批真实协议做 PoC 验证。低置信度字段强制人工复核。每个字段单独打分,不达标的字段不上线。
中风险

跨域数据一致性

Task 域、Knowledge 域、Extraction 域产生大量交叉引用。一处更新,多处需同步。

对策: 用领域事件机制 (Domain Events) 做最终一致性。Task 域作为事件源,其余域订阅。审计层强制要求所有跨域操作有事件留痕。
中风险

Phase 3 工具调用的合规边界

金融环境下,Agent 自主调用工具一旦出错,责任归属不清。"AI 自主决策" 在受监管行业是敏感词。

对策: Phase 3 一定要 Compliance 提前介入。工具调用永远 "用户在 loop 中",每一步必须用户确认。审计链覆盖每一次工具调用。绝不做对客户账户的写操作。
中风险

飞轮回流质量失控

任务产物自动注册回知识空间,如果产物本身有错(LLM 幻觉),会污染知识资产,放大错误。

对策: 产物注册必须经过 Maker-Checker 审核流。建立 "知识可信度" 标签,标识哪些是经审核的、哪些是自动沉淀的。检索时按可信度加权。

守住三件事就能走稳: 术语严格,合规先行,小步快跑。
