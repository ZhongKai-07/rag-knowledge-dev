§ 01

问题与起点

为什么我们必须从 "知识库问答" 升级到一个新的产品形态。

我们最初的命题是: 用 RAG 为 Ops 各团队搭建一个智能问答系统。但随着业务深入,我们遇到了三个 RAG 范式无法承载的真实需求——它们让我们不得不重新思考产品的形态边界。

图 01从单一 RAG 命题到四类真实业务场景的演化

这四个场景的形态相似但本质不同:输入形态(自然语言 vs 文档)、输出形态(文本 vs 结构化数据)、执行模式(单轮检索 vs 确定性 Pipeline vs Agent loop)、失败成本(重新问 vs 法律风险)、可复用性(一次性 vs 模板化)——五个维度上几乎全不一样。如果我们继续用 "Chatbot 加几个功能" 的方式去 cover,只会越做越乱。

真正的产品形态,是 Ops 同事每天工作里的 "任务" 本身——不是 "问答",也不是 "Agent"。

§ 02

业内启发

设计这套架构不是凭空而来。它根植于业内最近两年关于 AI、知识与 Agent 的几个核心命题。这一章梳理给我们带来根本性启发的四个洞察,以及它们如何转化为我们的设计原则。

这四个启发汇聚成一个判断: 下一代企业 AI 产品,不再是 "把 AI 接入系统",而是 "把系统建立在 AI 之上"。我们要做的,是一个 AI 原生的工作台,不是给 RAG 加几个功能。

INSIGHT 01
LLM Wiki · 知识沉淀派

RAG 不该每次从零检索

RAG 的根本缺陷,是把每一次问答都当作 "从零开始的检索任务"——知识不沉淀、不结晶、不升级。理想形态是让 LLM 真正 "懂" 知识库,而不是每次重新翻一遍语料。高频问答应该被验证、被收敛、成为 canonical 知识,下一个相同的问题不再走 RAG,而是直接命中已沉淀的答案。
→ 给我们的指导
任务产物必须有反向沉淀通道(知识飞轮)。审核通过的高频 Q&A 升级为知识资产,而不是埋在对话日志里。RAG 是兜底机制,不是唯一机制。
INSIGHT 02
Claude Code Memory · mem0

分级记忆是 Agent 的下半场

上半场拼模型能力,下半场拼上下文工程。让 Agent 懂用户、懂场景、懂历史,核心是分级记忆——短期工作记忆(本次任务上下文)、中期 episodic 记忆(用户历史行为)、长期语义记忆(空间共享知识)。这是上下文工程的精髓,也是 Agent 从 "能用" 到 "好用" 的分水岭。
→ 给我们的指导
三层记忆架构: 任务级 / 用户级 / 空间级。上下文不是单次 prompt,是持续构建、持续升级的资产。Memory 严格按 Space 隔离,符合金融合规要求。
INSIGHT 03
ChatGPT Pulse

从被动响应到主动洞察

Pulse 把 "用户问 → AI 答" 的被动模式,改造为 "AI 主动洞察 → 用户响应" 的主动模式。AI 不再是一个永远等待提问的工具,而是一个会主动发现、主动提示的同事。这代表了 AI 助手交互范式的根本转变——从工具变成同事。
→ 给我们的指导
系统的终极形态包含主动 Pulse,但金融场景必须严格分级: 先做内部观察类(本周高频咨询、知识缺口报告),不碰对客建议。Pulse 排在 Phase 4,是远期能力。
INSIGHT 04
业内对 AI 开发本质的共识

AI 替代了原本的 if-else

AI 开发的本质,是用大模型在预训练阶段固化的知识,加上人类在后续提供的内容(post-training、上下文工程、提示工程),去解决现实里 70% 原本必须用 if-else / 规则引擎硬编码的"软逻辑"问题。从 Pretrain → Post-train → Context → Prompt,本质上是把规则塑造的责任,从程序员手里转交给一条知识与提示的流水线。
→ 给我们的指导
不再用传统硬编码思维做产品。规则、判断、流程交给 LLM 承载,通过 schema、提示、上下文、记忆塑造行为。这是 AI native 思维,也是这套架构和传统软件架构最大的差异。
图 02四个业内启发如何汇聚为四条设计原则,共同支撑 AI 原生工作台
AI-NATIVE MANIFESTO
用 AI 思维 重构系统,不是给系统 加 AI
这四条启发放在一起看,我们要做的事就清晰了: 一个有持续记忆、有主动能力、有知识沉淀、用 LLM-native 方式承载业务逻辑的工作台。它不是 RAG 系统的升级版,是另一个物种。这意味着团队在做技术选型和功能拆解时,需要从 "在已有架构上加 AI 能力" 的思路,切换到 "AI 是一等组件,围绕它组织整个系统" 的思路。

这种切换不是术语层面的差异,而是设计动力学的差异。传统系统是 "用代码定义行为",LLM-native 系统是 "用上下文塑造行为"; 传统系统的智能在编译期固化,LLM-native 系统的智能在运行时被持续注入(通过 schema、prompt、记忆、检索)。这意味着我们的代码量会减少,但提示词、模板、上下文资产的设计量会大幅增加——这恰恰是接下来五大原语和工作台层的核心工作。

§ 03

产品定位:三个"不是",一个"是"

在金融受监管环境下,我们要做的产品有清晰的边界和反边界。

Not 01

不是
智能知识库

RAG 只是底层能力之一。我们要承载的任务远不止 "问答",还有抽取、混合检索、复杂指令、问题排查。
Not 02

不是
Agent 工厂

我们不像 Coze / Dify 让用户自由拼装 Agent。受监管环境下的产品逻辑是 "任务先行,Agent 是任务的执行形态之一"。
Not 03

不是
横向 Copilot

不像 Microsoft Copilot / Gemini Enterprise 在已有数据上叠 AI。我们主动培育和治理知识资产,做合规可审计的运营平台。
What we are

面向业务任务的
结构化工作台

把 Ops 真实工作流里的 "任务" 抽象为产品的一等公民。不同任务有不同执行原语、不同 UI、不同产物,但共享底层能力,统一治理与沉淀。

§ 06

工作台清单

用户面对的不是一个 Chatbot,也不是一个 Agent 列表,而是一组面向具体任务的工作台。每个工作台有独立的 UI、独立的产物管理、独立的评估指标。

P0 · 现有改造

知识问答工作台

承载 COB / SOP 等基于稳态知识的 NL 查询。延续 Chatbot 形态,但作为整体工作台体系下的一个实例。
  • 对话式交互,流式返回
  • 引用回链与置信度提示
  • 多轮上下文与会话管理
  • 高频 Q&A 沉淀入口
P0 · 新建

文档抽取工作台

承载协议智能提取场景。面向任务的工作流型 UI,与对话流截然不同。
  • 模板中心 (ISDA / CSA / GMRA)
  • 批量上传与任务队列
  • 字段级置信度 + 原文回链
  • Maker-Checker 复核流
  • Excel / JSON 导出 + 版本管理
  • 产物可注册回知识空间
P0 · 新建

混合筛查工作台

承载 Colleteral 智能筛查。表面像问答,实则是 "检索 + 字段抽取" 的复合产品。
  • 查询条件结构化 (交易对手、字段)
  • 结果以表格 + 引用呈现
  • 命中字段可批量导出
  • 不命中时提示降级到全文检索
  • 复用文档抽取的字段 schema
P2 · 未来

复杂指令工作台

承载需要 Agent loop 的多步任务。规划在 Phase 3 落地,合规设计先行。
  • NL 指令 → 可视化执行计划
  • 每步用户确认与中止
  • 受控 Tool Registry
  • 完整审计链 + 回滚
  • 任务模板可复用
P3 · 未来

问题排查工作台

承载诊断类任务。重点不是给答案,是给排查思路。
  • 异常现象输入 + 上下文上传
  • 假设链可视化
  • 历史案例匹配
  • 逐项验证清单
  • 明确 "辅助" 定位与免责
横向 · 后台

知识管理后台

所有工作台的共享后台。承担文档治理、产物归档、知识沉淀。
  • 多 Space 多租户
  • RBAC + ABAC + ACL 权限
  • 文档生命周期管理
  • 任务产物归档与回流
  • 沉淀的 Q&A / 模板治理
  • 合规审计与链路追踪
§ 07

知识飞轮:任务产物 → 知识资产

这套架构最关键的,是任务执行的高质量产物能反向沉淀回知识空间,形成持续运转的飞轮。这是和 Coze / Copilot 类产品的根本差异。

不是 "回答一次问题",而是 建立一个可持续运转的知识闭环。任务产生数据,数据沉淀为资产,资产反哺下一次任务。

图 03知识飞轮的四步闭环

这个闭环里有两个关键的反向通道是 Coze / Copilot 类产品都不具备的: 第一是任务产物注册回知识空间(高质量字段成为可被检索的资产); 第二是任务模式沉淀为模板/skill(下次同类任务无需从零做起)。

需要警惕的是,任务过程数据(执行轨迹、轨迹日志)和任务产物(结构化字段、验证过的答案)是两个完全不同层次的概念——前者用于观测和调优,后者才是飞轮的真正燃料。两者不能混为一谈。

§ 08

差异化定位

在和市面同类产品对比时,我们的护城河在哪里。

COZE / DIFY 类

Agent 装配平台

他们的逻辑
自下而上,每个用户/团队搭自己的小 Agent + 小知识库。
结构性缺陷
没有强制治理层,知识资产碎片化、不可信、不可审计。
COPILOT / GEMINI ENTERPRISE

横向 AI 助手

他们的逻辑
在已有数据上叠 AI,索引邮件、文档、日历做横向搜索 Q&A。
结构性缺陷
几乎不主动治理知识,知识质量等于企业现有数据质量。垃圾进垃圾出。
OURS

任务化平台

我们的逻辑
业务任务为一等公民。任务驱动产物,产物沉淀回知识空间,知识反哺下一次任务。
结构性优势
合规可审计 · 知识资产持续运营 · 受监管行业可用 · 任务和资产双向飞轮

用一句话表述差异化:在受监管行业里,以 "知识资产 + 任务能力" 双重持续运营为中心,把 RAG / Agent / Memory 统一在一个可治理的闭环里。

我们不是要做 "更好的 Coze",也不是 "金融版的 Copilot",而是一个新的产品品类。把这点对内对外讲清楚,是这份文档存在的根本原因。

§ 09

统一术语

团队对内沟通的标准词汇表。混用会导致设计偏移,请严格使用。

任务
Task
用户在工作台发起的一次明确目的的执行单元。是平台的一等公民,有 ID、状态、输入、产物、轨迹。所有业务需求都用 Task 描述。
任务原语
Task Primitive
任务的执行模式抽象,共五种: Retrieval QA、Extraction、Hybrid、Multi-step、Diagnostic。一个原语对应一个 Executor。
任务执行器
Task Executor
实现某个原语的后端组件。比如 RetrievalQAExecutor、ExtractionExecutor。对外不要叫 "Agent"。
Agent
Agent (narrow)
仅指具备自主规划能力的执行器(Multi-step、Diagnostic)。RAG 和 Extraction 不是 Agent易混词
工作台
Workbench
面向用户的产品形态。每个工作台对应一个任务原语,有独立 UI、独立产物管理。
任务产物
Task Artifact
任务产生的高质量结果(结构化字段、验证过的答案、生成的 Excel 等)。可注册回知识空间。不等于任务的执行轨迹日志。
任务轨迹
Task Trace
任务执行过程的中间步骤记录。用于观测、调试、评估。不直接进入飞轮。
知识空间
Knowledge Space
租户级的知识容器,承载文档、Q&A、模板、字段等知识资产。受 RBAC + ABAC + ACL 权限模型管控。
沉淀
Crystallize
把高质量任务产物注册为可复用知识资产的过程。是飞轮的核心动作。
原子能力
Capability Atom
底层共享的基础能力(检索引擎、抽取引擎、文档解析、工具调用、评估)。多个原语共用。
飞轮
Flywheel
"任务 → 产物 → 沉淀 → 反哺任务" 的闭环。是产品的核心动力学。非营销用语,是真实的架构机制。
