II — Product View 我们在解决什么

一个问题， / 一份带出处的依据。

业务同学每天都在文档堆里"找一句话"。HT KnowledgeBase 把"找文档 → 读文档 → 给答案"压缩成一次对话，并且每一句答案后面都跟着可点击的引用。

i.

谁在用

业务、产品、客服、管理者，按部门入座。每个人看到的是被授权的那部分知识。

ii.

在做什么

选一个知识空间 → 提一个问题 → 拿到答案 + 引用 + 后续追问建议。

iii.

痛点

文档散乱、术语不统一、问得不准。传统检索给链接，不给答案。

iv.

价值

答案带引用 → 可追溯；权限分级 → 可审计；评估闭环 → 可持续迭代。

业务诉求产品功能背后的技术能力不变量
"我只想问，不想翻文档"对话式问答 / 流式输出查询改写 + 多路召回 + LLM 生成每句答案附 [^n] 引用
"答案不能编"引用卡片 / 来源回链检索结果 → Prompt 注入 → 引用 ID 对齐引用必须可点回原 chunk
"我不能看见别人的资料"知识空间 / 部门隔离RBAC + KB 绑定 + 检索阶段过滤无授权即拒绝（fail-closed）
"答得好不好要量化"评估管理 / 黄金集RAGAS + 评估采集 + 离线运行每次回答都被采样可追溯
III — Personae 三种角色，一道围栏

谁在场， / 谁能看到什么。

权限围绕"知识空间"展开 —— 一个 KB 即一道隔离边界，kb_id 同时锁住会话、检索结果与 UI 入口。

I
USER · 普通用户

问答者

在被授权的知识空间里提问、追问、收藏答案。
  • 只能进入分配给自己角色的 KB
  • 问答历史按 KB 隔离，跨空间不可见
  • 看到的引用 = 自己有权读到的 chunk
II
DEPT_ADMIN · 部门管理员

看护者

管理本部门的知识空间、文档与角色，不能跨部门越线。
  • 建/改 KB；上传 / 重切 文档
  • 分配本部门用户到角色
  • 查看本部门的问答追踪与评估
III
SUPER_ADMIN · 超管

设计者

全局：模型路由、意图树、评估套件、部门与角色拓扑。
  • 配置 LLM / Embedding / Rerank 候选与降级
  • 编辑意图树（域 → 类目 → 主题）
  • 跑评估、看大盘、定基线
隔离维度实现位置不变量
会话归属t_conversation.kb_id + 前端 URL ?kbId=无 kbId 不能开会话；旧会话 NULL 视为不归属任何空间
检索范围OpenSearch index 按 collectionName 一对一问答只查当前 KB 的 index，不会"串味"
结果可见性AuthzPostProcessor（重排前）无权访问的 chunk 在召回结果中直接剔除

V — Ingestion 文档入库流水线

从一份 PDF / 到一行可被检索的向量。

上传是同步的（你立刻看到状态），但解析、切块、向量化是异步的：通过 RocketMQ 投递任务，节点链式处理，状态机贯穿全程。

i.

解析 Parse

Apache Tika 3.2 兼容 PDF / Word / Excel / PPT / HTML / MD；docling 增强方案规划中。

ii.

切块 Chunk

默认按字符固定长度（1000 字 + 重叠对齐边界），结构感知策略支持 Markdown 标题层级。

iii.

向量化

infra-ai 的 RoutingEmbedding：多家候选（百炼 / 硅基 / Ollama），按优先级路由 + 故障降级。

iv.

状态机

PENDING → RUNNING → SUCCESS / FAILED；每阶段耗时落 chunk_log，便于故障定位。

VI — QA Pipeline 问答六步走

六个动作， / 一气呵成。

用户的一句话进来后，编排器按固定节奏依次完成六件事；每一步都被同一个 traceId 串起来，所有耗时与产物都可被审计。

i
记忆补全

取最近若干轮对话 + 可选摘要，把"指代"还原成完整问题。

ConversationMemoryService
ii
查询改写

规则归一化术语 → LLM 改写并拆分成子问题（多查询展开）。

MultiQuestionRewriteService
iii
意图分类

对照"域 → 类目 → 主题"三层意图树打标，附置信度。

IntentResolver / DefaultIntentClassifier
iv
多路召回

意图导向 + 全局向量 + MCP 工具，三路并行。

RetrievalEngine + MultiChannelRetrievalEngine
v
后处理

权限过滤（fail-closed）→ 去重 → 重排选 top-K。

AuthzPostProcessor / Dedup / Rerank
vi
Prompt + LLM

注入 chunks + 引用规则 → 流式生成 → 答案 + 引用卡片。

RAGPromptService → LLMService.streamChat()
事件触发条件结果
歧义引导（短路 1）所有候选意图置信度都低不召回，直接返回澄清提示让用户选
系统意图（短路 2）问候 / 闲聊 / 帮助类系统意图命中跳过召回，直答 LLM
权限不足（fail-closed）用户对该 KB 或 chunk 的安全级无权结果在重排前剔除；若空则兜底拒答
评估采样命中采样率问题 + 改写 + chunks + 答案 落 t_rag_evaluation_record
核心约束 · 编排器是唯一下决策的地方；六个能力组件只回答"我能做什么"，不决定"下一步怎么走"。
VII — Key Modules 关键模块详解

把六步 / 各拆一层。

挑出最影响"答得准不准"的四个模块，看它们各自在做什么、有什么旋钮可调。

为什么要改写？

用户问的话往往不完整：用代词（"它"），用别名（"百炼" vs "DashScope"），或者一个问题套多层意图（"先帮我查 A，再对比一下 B"）。直接拿原话去检索，命中率会非常糟。

怎么改？

  • 规则归一化 · QueryTermMappingService：把同义词、缩写映射成标准表述。
  • LLM 改写 + 拆分 · MultiQuestionRewriteService.rewriteWithSplit()：结合最近对话历史，把"它怎么用？"改成"百炼平台的 SDK 怎么调用？"，并把多意图拆成子问题。
  • 规则兜底 · 改写失败时退回简单分句，保证下游能拿到至少一个查询。

能调什么

  • rag.query-rewrite.enabled · 全局开关
  • QUERY_REWRITE_AND_SPLIT_PROMPT_PATH · 改写提示词
  • 历史轮数（影响指代消解的上下文广度）

典型收益

同样的问题，开启改写后召回 Top-K 命中率显著提升；多查询展开能让"复合问题"也拿到完整证据。

三层意图树

意图按"域 Domain → 类目 Category → 主题 Topic"组织成一棵树（持久化在 t_intent_node，Redis 缓存）。

  • Domain · 一级域 e.g. 业务咨询 / 数据查询 / 系统功能
  • Category · 二级类目 e.g. 业务咨询/产品规格
  • Topic · 叶子主题，落到具体 KB / 工具

怎么打标

分类器（DefaultIntentClassifier）一次性把整棵意图树发给 LLM，让它返回多个候选意图 + 置信度。然后按 INTENT_MIN_SCORE 阈值过滤、按数量上限截断。

低置信度怎么办

  • 歧义引导 · 候选都低 → IntentGuidanceService.detectAmbiguity() 触发澄清，前端弹"你想问的是哪一类？"
  • 系统意图短路 · 命中"问候 / 帮助 / 闲聊" → 跳过 RAG 直接答
  • 意图合并 · 同时命中 KB 意图与 MCP 意图 → 各走各路并行检索

能调什么

  • 意图树：超管在意图树编辑页面增删节点
  • 置信度阈值 / 候选数量上限
  • 采样问题：把真实问题挂回意图节点，下次更准

三路并行

  • 意图导向通道 · 在命中意图绑定的 KB / 文档范围内做向量检索，召回更聚焦
  • 全局向量通道 · 在 KB 全量内做 ANN，避免意图打偏漏召
  • MCP 工具通道 · 命中 MCP 类意图（如"查抵押品字段"）时，调用注册的工具拿结构化结果

三路用 CompletableFuture 并发执行（30s 超时），互不阻塞。

三路汇合

  • 权限过滤 · AuthzPostProcessor（order=0）剔除用户无权读到的 chunk
  • 去重 · 基于 chunk hash，避免同一段被多路重复召回
  • 重排 · 召回池 30 → 重排选 Top-10（rerank-top-k 可配）

能调什么

  • recall-top-k · 召回池大小
  • rerank-top-k · 进 Prompt 的 chunk 数
  • 各通道开关（意图通道 / 全局通道 / MCP）

引用怎么生成

Prompt 里把 Top-K chunk 编上 [1] [2] [3] ...，并写明"必须用 [^n] 形式标出引用"。LLM 流式吐 token 时就把引用编号嵌入答案文本。

前端怎么显示

  • 答案中 [^1] 渲染成可点击的小角标
  • 下方引用卡片列表对应每个编号 → 文档名 / 段落片段 / 原文跳转
  • 持久化在 t_message.sources_json（v1.9 加入）

评估闭环

每轮问答都会按采样率把"问题 / 改写 / 召回 chunks / 最终答案"落进 t_rag_evaluation_record。可以：

  • 导出为 RAGAS 兼容 JSON · GET /rag/evaluations/export
  • 用百炼当评估器跑离线指标（faithfulness / answer relevancy / context precision）
  • 前端"评估管理"页面看运行结果、对比历史

追踪

RagTraceContext 用 TTL 跨线程透传 traceId，每个节点 @RagTraceNode 自动记录耗时与 token 用量到 t_rag_trace_node。

VIII — Multi-channel Retrieval 三路并行 · 殊途同归

三股水， / 汇成一杯茶。

为什么不只走一条最聪明的路？因为没有"最聪明"。意图通道精准但易偏，全局通道兜底但稀释，MCP 通道结构化但只对部分意图有效 —— 三路互补。

设计哲学 · 多路召回不是"打散注意力"，而是用不同视角互相校验。意图通道说"这事归我管"，全局通道说"我也看到一段相关的"，MCP 通道说"我有一份结构化数据"—— 拼起来才是完整答案。
