**总体判断**
从金融公司技术架构师视角看，这套知识库已经不是“从 0 到 1”的阶段了，能力框架其实搭得不错：有知识空间隔离、RBAC 骨架、`security_level`、异步分块、定时刷新、多通道检索、向量后端抽象。  
但它现在更接近“内部试点可用的平台雏形”，还不是“可承载高敏感金融知识资产”的生产级底座。我会把成熟度定在 `2.5/5`：业务能力是 `B`，安全治理和合规运营只有 `C-/D`。

**关键风险**
- `P0` 密钥治理失守。仓库配置里直接保留了数据库、Redis、S3、外部 LLM 的凭据和 API Key，[application.yaml](</E:/AIProject/ragent/bootstrap/src/main/resources/application.yaml:18>)、[application.yaml](</E:/AIProject/ragent/bootstrap/src/main/resources/application.yaml:121>)、[application.yaml](</E:/AIProject/ragent/bootstrap/src/main/resources/application.yaml:190>)。更严重的是，`/rag/settings` 会把 `apiKey` 原样返回，而全局网关只做登录校验，[RAGSettingsController.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/controller/RAGSettingsController.java:63>)、[SaTokenConfig.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/user/config/SaTokenConfig.java:57>)。这在金融场景属于直接不合规。
- `P0` 入库控制面仍然是高风险暴露面。入库 Pipeline/Task 控制器没有看到显式 RBAC，[IngestionPipelineController.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/controller/IngestionPipelineController.java:52>)、[IngestionTaskController.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/controller/IngestionTaskController.java:57>)；同时 URL 抓取链路直接接受外部地址并发起 HTTP 请求，没有看到私网/IP/域名白名单之类的出网控制，[HttpUrlFetcher.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/strategy/fetcher/HttpUrlFetcher.java:48>)、[HttpClientHelper.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/ingestion/util/HttpClientHelper.java:52>)。对金融机构来说，这相当于把 SSRF 和未经审批的数据引入入口暴露出来了。
- `P0` 评测与链路观测数据没有按用户/KB 隔离。控制器未加权限限制，[RagEvaluationController.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/controller/RagEvaluationController.java:48>)、[RagTraceController.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/controller/RagTraceController.java:47>)；服务查询也没按 `userId` 或 `kbId` 过滤，[RagEvaluationServiceImpl.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/impl/RagEvaluationServiceImpl.java:77>)、[RagTraceQueryServiceImpl.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/impl/RagTraceQueryServiceImpl.java:63>)。而这些表里存的是 query、retrieved chunks、answer、trace 内容，本质上就是敏感知识访问痕迹。
- `P0` `security_level` 的“变更后刷新向量元数据”链路很可能是失效的。发送端把 topic 写成了普通字符串 `"knowledge-document-security-level_topic${unique-name:}"`，[KnowledgeDocumentServiceImpl.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/service/impl/KnowledgeDocumentServiceImpl.java:583>)；producer 只是把 topic 原样发给 RocketMQ，并不会解析占位符，[RocketMQProducerAdapter.java](</E:/AIProject/ragent/framework/src/main/java/com/nageoffer/ai/ragent/framework/mq/producer/RocketMQProducerAdapter.java:45>)；而消费者注解里的 topic 会由 Spring 解析，[KnowledgeDocumentSecurityLevelRefreshConsumer.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/mq/KnowledgeDocumentSecurityLevelRefreshConsumer.java:37>)。据此推断，文档安全级别变更后，OpenSearch 侧 metadata 可能根本没同步，授权会出现“库里改了、检索侧没改”的不一致。
- `P1` 向量后端抽象在“能力接口”上统一了，但在“授权语义”上并不一致。OpenSearch 路径会带 `security_level` filter，[MultiChannelRetrievalEngine.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/MultiChannelRetrievalEngine.java:275>)；但 Milvus/pg 检索实现没有用 `metadataFilters`，[MilvusRetrieverService.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/MilvusRetrieverService.java:68>)、[PgRetrieverService.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/PgRetrieverService.java:54>)，写入实现也明确写了 `kbId/securityLevel ignored` 或 `updateChunksMetadata unsupported`，[MilvusVectorStoreService.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/vector/MilvusVectorStoreService.java:59>)、[PgVectorStoreService.java](</E:/AIProject/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/vector/PgVectorStoreService.java:45>)。这意味着“切换向量后端”不是纯基础设施切换，而是授权模型变化。
- `P1` 控制制度还没固化成“默认拒绝”。我没有在代码里看到 `RequiresPermission`、`PublicEndpoint`、`ArchUnit` 或访问审计表落地；而现有测试里也基本只有 `@SpringBootTest`，没看到 `MockMvc`/Sa-Token 的接口授权回归测试。对金融场景，这意味着当前安全更多依赖人工自觉，而不是工程制度。

**建议路线**
先别把它定义成“金融级知识库平台”，更准确的说法是“具备金融知识服务雏形的 RAG 平台”。下一阶段应该把重心从“能不能用”切到“能不能被审计、被治理、被证明安全”。

建议按这个顺序推进：
- `30 天止血`：撤出源码中的所有密钥；关闭 `/rag/settings` 的敏感字段暴露；给 ingestion/trace/evaluation 控制面补权限；修掉 `security_level` MQ topic 问题；给 URL 抓取加内网地址和域名白名单。
- `60 天固化`：建立默认拒绝的权限框架和 ArchUnit/接口回归测试；把访问拒绝和敏感操作落审计；让 DEPT_ADMIN、KB、security_level、trace/eval 形成统一的授权矩阵。
- `90 天达标`：补数据分级流程、文档引入审批、恶意文件扫描/DLP、KMS/密钥轮换、留痕保留策略、容灾与容量基线。做到这一步，才像金融公司的“知识资产平台”而不是“RAG 应用”。
