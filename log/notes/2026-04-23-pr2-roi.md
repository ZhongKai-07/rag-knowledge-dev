# PR2 相比 PR1 的正向收益

> 触发：PR2 Step 9.9 真实回归后想清算一下这个 PR 的投入产出。

## 直接可测量的数字（本次 session）

| 指标 | pre-PR1 | PR1 | PR2 |
|------|---------|-----|-----|
| "高风险客户..." maxScore | 0.5（stuck） | 0.8119 | 0.8541 |
| "地球到太阳..." maxScore | 0.5（stuck） | 0.2187 | 0.2705 |
| bailian 实际候选池 | — | `dedup=10, topN=10` | `dedup=30, topN=10` |
| rerank 的选择空间 | 0 条 | 0 条（仍=10:10） | **20 条**（30→10） |

maxScore 单点只涨 +4%（0.81→0.85），但这只是 PR1 已经吃掉主要增益之后的**额外**收益。真正的质变在 **rerank 从"没得选"变成"有 20 条备胎可换"**。

## PR1 vs PR2 的本质差别

- **PR1**：让 cross-encoder **跑起来**了（之前被 `candidates.size() <= topN` 短路），但**给它的候选池还是 bi-encoder 挑的那 10 条**。= "找了个专家复审，但专家只能在原班 10 条里重新排序，不能换人"。
- **PR2**：候选池拓宽到 30，rerank **可以淘汰 20 条换成更好的**。= "专家有 30 条可看，最终挑 10 条。bi-encoder 错把好 chunk 排在第 25 位？PR2 能救，PR1 不能"。

## 结构性收益（单次 maxScore 看不到的）

| 维度 | pre-PR2 | PR2 |
|------|---------|-----|
| 候选池配置 | `DEFAULT_TOP_K × topKMultiplier` 暗算（10×2 / 10×3），双 channel 不一致 | 一行 yaml `recallTopK=30`，双 channel 对称 |
| 通道内 trim | 隐式、各 channel 实现不同 | 显式 `fan-out N -> cap recallTopK`，日志可见 |
| 启动期校验 | 无 | `@PostConstruct` 验 `recallTopK ≥ rerankTopK > 0`，错配 fail-stop |
| 运行期校验 | 无 | `SearchContextBuilder.build()` 再兜一道 IAE |
| 评测字段 | `setTopK(DEFAULT_TOP_K=10)`—**说谎**（override / dedup 场景） | `setTopK(distinctChunks.size())`—ground truth |
| 小 KB 边界 | 靠百炼 API 未定义行为 | `min(topN, candidates.size())` 客户端兜底 |
| IntentNode.topK 语义 | 混乱（"最终数" vs "召回数"？） | 明确"最终保留数"，retrieval 层翻译成 rerank 端 |

## 代价

- **代码规模**：+606/-116 行，19 个文件
- **新增**：1 个 `@ConfigurationProperties` bean、1 个 `record`、4 个新测试、3 个已有测试迁移
- **向量库查询量**：IntentDirected 单次原始候选从 `10×2=20` → `30`（+50%）；VectorGlobal `10×3=30` → `30`（持平）
- **百炼 rerank 输入 token**：10 条 chunks 文本 → 30 条（**~3×**）。rerank 是 RAG 中最便宜一环（¥几分钱/千次），3× 无感。延迟估计 +100~200ms。

## 还没测但值得做的

- **RAGAS recall@10 / precision@10**：理论上 PR2 应在 recall 端有明显涨幅（bi-encoder 漏在第 11-30 位的文档现在有机会被 rerank 捞回来）
- **latency 真实分布**：今天 session 两次问答都 <1s，样本太小
- **IntentNode.topK override 路径**：`effectiveRerank = node.topK, effectiveRecall = max(global, node.topK)` 需要一个配了 override 的意图节点做真实回归

## 一句话结论

**PR1 让 cross-encoder 发声；PR2 让它能改变结果。**

maxScore 单点 +4% 是因为 PR1 已经救了主要 case。PR2 的真实价值在：
1. 给 rerank 20 条备胎换人权
2. 整条链路从"隐式 trim"变成"显式 sort+cap+配置化"
3. 评测口径归真
4. 从"能跑"升级成"可调可观察可信任"

标准 RAG over-retrieve 架构从此归位。
