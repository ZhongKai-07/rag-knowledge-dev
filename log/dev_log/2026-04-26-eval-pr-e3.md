## 关键决议

1. **不抽 AnswerPipeline**（ADR 决议）—— `ChatForEvalService` 直接复用 6 个现有 RAG service bean；`streamChat` 字节级不变；`@ChatRateLimit` 不挂在 eval 路径。
2. **EVAL-3 与 E3 同 PR**（用户决议）—— 任何返回 `retrieved_chunks` 的 API 必经 `EvalResultRedactionService`；列表 / 趋势 VO 不含 `retrievedChunks` 字段（契约即门禁）；当前 SUPER_ADMIN-only + ceiling=`Integer.MAX_VALUE`，未来放 AnyAdmin 仅需切 ceiling。
3. **三态状态机**取代旧"≥1 即 SUCCESS" —— 1/50 显示绿灯会误导趋势判断。`failed==0 && succeeded>0` → SUCCESS，`succeeded==0` → FAILED，否则 PARTIAL_SUCCESS。
4. **`max-parallel-runs=1` 硬 enforce**（review P1-4 升级）—— `ReentrantLock startRunLock` 包住 SELECT COUNT + INSERT + executor.execute；不再依赖软校验。8 线程并发测试锁此语义。
5. **`AccessScope` 由调用方注入**（review P1-1）—— `ChatForEvalService.chatForEval(scope, kbId, q)` 永不自造 scope；唯一合法 `AccessScope.all()` 持有者是 `EvalRunExecutor`（spec §15.3）。
6. **`config_hash` 仅基于 RAG-behavior 字段**（reviewer 修订）—— `git_commit` 项目无 maven plugin 永远是 `"unknown"`，移到 hash 之后做元数据；canonical form 用 compact JSON（不是 pretty-print）规避 Jackson 升级 hash 漂移。
7. **DTO field rename `goldItemId` → `resultId`**（reviewer 修订）—— Java→Python `/evaluate` 字段实际carries `EvalResultDO` snowflake（不是 GoldItemDO id），便于 batch 内多条同 goldItem 精确回填指标；rename 在 T9 fix 提前到 T13 Python 实现之前，避免 wire 契约带误导名。

## 实施期遇到的偏差

- **`EvalRunExecutor` interface→class 替换**：T8 创建占位 interface，T9 替换为 @Service class。Spring 按类型注入，`EvalRunServiceImpl` 无需改动，但首次 mvn 编译时 ServiceImpl 会短暂 broken（T9 完成后恢复）。
- **`AIModelProperties.ModelGroup` 单一内类**：plan 写 `Chat()/Embedding()/Rerank()` 内类不存在；测试用 `new AIModelProperties.ModelGroup()`（pre-fix-B）。
- **`SubQuestionIntent` 字段名 `subQuestion` 非 `question`**（T2 实施时发现）/ **`IntentNode` 包路径 `rag.core.intent` 非 `rag.intent.domain`**（plan 笔误，T2 修正）：均为 plan 笔误，实施时按真实 API 调整。
- **`RoutingLLMService.executeWithFallback` 是 4 参非 2 参**（T7b 实施时发现）：plan 写 `executeSyncWithFallback(2 args)` 不存在；按真实 4 参方法（capability, targets, clientResolver, caller）适配。
- **`sheet` 组件不在 UI 库**（T17 实施时发现）：plan 写 `Sheet/SheetContent`，frontend `components/ui/` 无此文件；改用 `Dialog` + `max-h-[80vh]` overflow-y-auto 模拟抽屉。
- **WireMock 依赖未在 bootstrap pom**：T7 实施时新加 `wiremock-standalone:3.10.0` test-scope（避免 transitive Jetty/Jackson 冲突）。
- **`langchain-openai` 未在 ragas requirements**：T13 实施时加 `langchain-openai==0.2.*`，对接 RAGAS 0.2.x `LangchainLLMWrapper` / `LangchainEmbeddingsWrapper`。
- **本地 ragas 包名 shadowing PyPI**：项目目录 `ragas/ragas/` 与 PyPI `ragas` 同名；T13 用 `_load_pypi_ragas_symbols()` 在 module load 时手工剥离 `sys.modules` shadowing 后导入 PyPI ragas API（评估、`EvaluationDataset`、4 个 metric singleton、`Langchain*Wrapper`）。
